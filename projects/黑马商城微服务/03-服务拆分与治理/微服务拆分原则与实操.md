# 微服务拆分原则与实操

## 一、写在前面

单体架构写完后，下一步就是拆微服务。这篇笔记解决两个问题：
1. 什么时候该拆、怎么拆（理论）
2. 实际动手拆商品模块和购物车模块（实操）

## 二、什么时候拆

| 阶段 | 策略 | 原因 |
|---|---|---|
| 初创/验证期 | 先单体 | 快速试错，成本低，发现不行损失小 |
| 用户量增长、业务变复杂 | 逐步拆微服务 | 单体扛不住了 |
| 大型项目立项 | 直接微服务 | 前难后易，省得后期痛苦拆分 |

**核心逻辑**：先单体快速验证市场，等业务复杂了再拆。小型项目"前易后难"，大型项目"前难后易"。

## 三、怎么拆——两个原则

### 3.1 高内聚

每个微服务的职责要尽量单一，内部业务关联度高、完整度高。

> 目标：修改某个业务时，最好只改当前微服务，变更成本最低。

**注意**：高内聚 ≠ 一个服务就一个接口，而是保证服务内部业务的完整性。

### 3.2 低耦合

每个微服务的功能要相对独立，尽量减少对其他微服务的依赖。

**关键做法**：不直接访问其他服务的数据库，通过接口交互。

```
❌ 订单服务直接查商品数据库（数据耦合）
✅ 商品服务暴露接口，订单服务调用接口（接口耦合，可接受）
```

虽然服务间调用不可避免，但只要**接口外观不变**，商品服务内部怎么改都不影响订单服务。

**因果关系**：高内聚做到了，低耦合自然就降低了。

## 四、怎么拆——两种方式

### 4.1 纵向拆分（按功能模块）

按业务功能切，提高内聚性：

```
黑马商城纵向拆分
├── 用户服务
├── 订单服务
├── 购物车服务
├── 商品服务    ← 先拆这个
└── 支付服务
```

### 4.2 横向拆分（抽取公共业务）

各模块共有的业务抽出来，提高复用性：

```
横向拆分示例
├── 消息中心服务（各模块都要发短信）
└── 风控管理服务（各模块都要记录风控数据）
```

横向拆分的公共业务一般接口稳定性强，不会使服务之间过分耦合。

## 五、实操：拆商品模块

### 5.1 新建 item-service 模块

在项目根目录新建 `item-service`，标准的 Spring Boot 项目结构：

```
item-service/
├── pom.xml
├── src/main/java/com/hmall/item/
│   ├── ItemApplication.java           ← 启动类
│   ├── controller/
│   │   ├── ItemController.java        ← 商品接口
│   │   └── SearchController.java      ← 搜索接口
│   ├── domain/
│   │   ├── dto/ItemDTO.java
│   │   ├── dto/OrderDetailDTO.java
│   │   ├── po/Item.java
│   │   └── query/ItemPageQuery.java
│   ├── mapper/ItemMapper.java
│   └── service/
│       ├── IItemService.java
│       └── impl/ItemServiceImpl.java
└── src/main/resources/
    ├── application.yaml
    ├── application-dev.yaml
    └── application-local.yaml
```

### 5.2 旧代码怎么处理

`hm-service` 里商品相关的代码（ItemController、ItemServiceImpl、ItemMapper 等）暂时保留不影响运行，但**最终一定要删**。

> 微服务拆分的核心：一个模块只由一个服务负责。两边都保留 = 复制了一份，不叫拆分。

### 5.3 购物车模块也一起拆了

同样新建 `cart-service`：

```
cart-service/
├── CartApplication.java
├── controller/CartController.java
├── domain/
│   ├── dto/CartFormDTO.java
│   ├── dto/ItemDTO.java
│   ├── po/Cart.java
│   └── vo/CartVO.java
├── mapper/CartMapper.java
└── service/
    ├── ICartService.java
    └── impl/CartServiceImpl.java
```

### 5.4 根 pom.xml 配置

根模块的 `pom.xml` 新增两个子模块声明：

```xml
<modules>
    <module>hm-common</module>
    <module>hm-service</module>
    <module>item-service</module>   <!-- 新增 -->
    <module>cart-service</module>   <!-- 新增 -->
</modules>
```

## 六、扣库存代码补充（executeBatch）

拆分过程中看到商品服务的扣库存实现，记录一下：

```java
@Override
public void deductStock(List<OrderDetailDTO> items) {
    String sqlStatement = "com.hmall.mapper.ItemMapper.updateStock";
    boolean r = false;
    try {
        r = executeBatch(items, (sqlSession, entity) ->
                sqlSession.update(sqlStatement, entity));
    } catch (Exception e) {
        throw new BizIllegalException("更新库存异常，可能是库存不足!", e);
    }
    if (!r) {
        throw new BizIllegalException("库存不足！");
    }
}
```

**关键点**：
- `executeBatch` 是 MyBatis-Plus `ServiceImpl` 提供的批量执行方法
- `sqlStatement` = 全限定方法名（包名.类名.方法名），MyBatis 靠这个定位 SQL
- Lambda 里的 `sqlSession.update(sqlStatement, entity)` = 对每个商品执行一次扣库存 SQL
- 对应的 SQL：`UPDATE item SET stock = stock - #{num} WHERE id = #{itemId}`（乐观锁写法，数据库层面做减法）

**Java 增强 for 循环**（for-each）：

```java
// 老写法
for (int i = 0; i < items.size(); i++) {
    OrderDetailDTO item = items.get(i);
}

// 新写法（效果一样，更简洁）
for (OrderDetailDTO item : items) {
    // item 就是当前遍历的元素
}
```

读作：从 items 里依次取出每个元素，赋值给 item。

## 七、全文总结

1. **什么时候拆**：先单体验证，业务复杂了再拆；大型项目可直接微服务
2. **怎么拆**：高内聚（职责单一）+ 低耦合（通过接口交互，不访问对方数据库）
3. **两种方式**：纵向按功能切（提高内聚），横向抽公共业务（提高复用）
4. **实操**：新建 `item-service` 和 `cart-service`，代码从单体搬过来，旧代码最终要删
