# 【Gateway】整体流程与核心组件

## 一、写在前面

1. 之前都是实操配置和写代码，这篇补上 Gateway 的理论框架——理解请求从进到出的完整生命周期
2. 学完这篇你能掌握：Gateway 内部是怎么运作的、过滤器链是怎么组装和执行的、PRE/POST 两阶段的含义

## 二、整体流程总览

客户端请求进入网关后，依次完成：

```
路由匹配 → 组装过滤链 → 前置过滤器执行 → 转发微服务 → 后置过滤器逆序执行
```

最终将处理后的响应返回客户端，形成一条完整的**请求进、响应出**的双向处理链路。

## 三、核心组件详解

### 1. 路由映射器 HandlerMapping

**默认实现：** `RoutePredicateHandlerMapping`

**核心职责：**
- 接收请求后，根据配置的断言规则（如路径、请求头、请求参数等）匹配目标路由
- 将路由信息存入请求上下文 `ServerWebExchange`
- 随后将请求转交 `WebHandler` 处理

### 2. 请求处理器 WebHandler

**默认实现：** `FilteringWebHandler`

**核心职责：**
- 加载网关全局过滤器 + 当前路由专属过滤器
- 按 `order` 优先级排序
- 组装成完整的过滤器链
- 驱动整条链按顺序执行

### 3. 过滤器 Filter：PRE & POST 双阶段逻辑

这是网关最核心的扩展能力，每个过滤器都包含**两段独立逻辑**：

| 阶段 | 执行时机 | 典型场景 |
|------|---------|---------|
| **PRE 前置逻辑** | 请求转发到微服务之前 | 身份鉴权、限流熔断、参数校验、请求头修改、请求日志埋点 |
| **POST 后置逻辑** | 微服务返回响应之后 | 响应头统一修改、响应体加工、跨域头注入、返回日志记录 |

**执行规则：**
- PRE 逻辑按 `order` **从小到大**顺序执行，全部通过后请求才会被转发
- 任意一个过滤器的 PRE 逻辑拦截请求，后续过滤器与转发流程都会终止
- 微服务返回结果后，POST 逻辑按 `order` **从大到小**倒序执行

### 4. Netty 路由过滤器 NettyRoutingFilter

它是过滤器链中负责**真正转发请求**的核心节点，也是 PRE 阶段的终点、POST 阶段的起点。

**核心职责：**
- 基于 Reactor Netty 客户端将请求转发到目标微服务
- 接收微服务响应并存入上下文
- 交由后续 POST 逻辑处理

## 四、执行顺序示例

假设有 3 个自定义过滤器，order 分别为 1、2、3，NettyRoutingFilter 优先级最低（order 最大）：

```
前置阶段：Filter1 (PRE) → Filter2 (PRE) → Filter3 (PRE) → NettyRoutingFilter（转发请求）

目标微服务处理业务并返回响应

后置阶段：Filter3 (POST) → Filter2 (POST) → Filter1 (POST)

最终响应返回至客户端。
```

**图示：**

```
请求进入
    ↓
┌─────────────────────────────────────────┐
│  1. HandlerMapping (路由映射器)          │
│     根据断言匹配路由，存入 ServerWebExchange │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│  2. WebHandler (请求处理器)              │
│     加载全局过滤器 + 路由过滤器，按order排序 │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│  3. PRE 阶段 (order 从小到大)            │
│     Filter1 → Filter2 → Filter3         │
│     • 鉴权、限流、参数校验、请求头修改     │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│  4. NettyRoutingFilter                  │
│     真正转发请求到微服务，接收响应         │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│  5. POST 阶段 (order 从大到小)           │
│     Filter3 → Filter2 → Filter1         │
│     • 响应头修改、跨域注入、日志记录       │
└─────────────────────────────────────────┘
    ↓
响应返回客户端
```

## 五、补充要点

### 过滤器分类

| 类型 | 说明 |
|------|------|
| **GlobalFilter** | 全局过滤器，对所有路由生效 |
| **GatewayFilter** | 路由专属过滤器，只对特定路由生效 |

运行时会被**合并、排序**后放入同一条执行链。

### 响应式编程

- 全流程基于 **Spring WebFlux** 响应式模型实现
- 底层依赖 **Netty**
- 全程**异步非阻塞**，适配高并发接入场景

### 上下文对象

`ServerWebExchange` 作为请求上下文贯穿全流程，所有组件都通过它传递请求、响应与附加业务数据。

## 六、对应你的代码

| 组件 | 你的实现 | 作用 |
|------|---------|------|
| **AuthGlobalFilter** | order = 0 | PRE 阶段：鉴权 + 传递 userId |
| **UserInfoInterceptor** | 微服务拦截器 | 接收 userId → ThreadLocal |
| **RequestInterceptor** | Feign 拦截器 | ThreadLocal → 下游请求头 |

## 七、全文总结

- Gateway 请求生命周期：路由匹配 → 组装过滤链 → PRE → 转发 → POST → 响应
- 核心组件：HandlerMapping、WebHandler、Filter、NettyRoutingFilter
- 过滤器分 PRE/POST 两阶段，执行顺序相反
- 全局过滤器和路由过滤器运行时合并排序
- 基于 WebFlux + Netty，全程异步非阻塞
