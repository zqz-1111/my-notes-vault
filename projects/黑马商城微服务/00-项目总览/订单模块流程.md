# 【订单】订单模块业务流程

## 一、写在前面

1. 订单模块是整个商城的核心，串联了商品、购物车、支付三个模块
2. 核心难点：一个方法里做了四件事，靠 `@Transactional` 保证原子性

## 二、创建订单（createOrder）

### 2.1 完整流程

```java
@Transactional
public Long createOrder(OrderFormDTO orderFormDTO) {
    // 1. 查询商品信息
    List<OrderDetailDTO> detailDTOS = orderFormDTO.getDetails();
    Map<Long, Integer> itemNumMap = detailDTOS.stream()
            .collect(Collectors.toMap(OrderDetailDTO::getItemId, OrderDetailDTO::getNum));
    Set<Long> itemIds = itemNumMap.keySet();
    List<ItemDTO> items = itemService.queryItemByIds(itemIds);

    // 2. 校验商品是否存在
    if (items == null || items.size() < itemIds.size()) {
        throw new BadRequestException("商品不存在");
    }

    // 3. 计算总价（price × num 累加）
    int total = 0;
    for (ItemDTO item : items) {
        total += item.getPrice() * itemNumMap.get(item.getId());
    }

    // 4. 写入订单表
    Order order = new Order();
    order.setTotalFee(total);
    order.setPaymentType(orderFormDTO.getPaymentType());
    order.setUserId(UserContext.getUser());
    order.setStatus(1);  // 1=未付款
    save(order);

    // 5. 写入订单详情表（批量）
    List<OrderDetail> details = buildDetails(order.getId(), items, itemNumMap);
    detailService.saveBatch(details);

    // 6. 清理购物车中已下单的商品
    cartService.removeByItemIds(itemIds);

    // 7. 扣减库存
    itemService.deductStock(detailDTOS);

    return order.getId();
}
```

### 2.2 四件事拆解

| 步骤 | 操作 | 调用的模块 |
|---|---|---|
| ① 查商品 + 算总价 | queryItemByIds | 商品模块 |
| ② 写订单 + 订单详情 | save + saveBatch | 本模块 |
| ③ 清购物车 | removeByItemIds | 购物车模块 |
| ④ 扣库存 | deductStock | 商品模块 |

### 2.3 事务保证原子性

```java
@Transactional  ← 这一个注解管住了四件事
public Long createOrder(OrderFormDTO orderFormDTO) { ... }
```

- 如果扣库存失败（第 7 步抛异常）→ 事务回滚 → 前面写的订单、订单详情、购物车清理全部撤销
- **单体架构下这个方案没问题**，因为所有操作在同一个数据库、同一个事务里

### 2.4 为什么先写订单再扣库存？

如果顺序反过来：

```
先扣库存 → 写订单失败 → 事务回滚 → 但库存已经扣了！
```

不对，事务回滚会把库存也回滚。那换个角度想：

```
先扣库存 → 扣成功了 → 写订单时发现商品已下架 → 抛异常回滚 → 库存也回滚了 → 没问题
```

其实顺序在单体架构下不影响（有事务兜底）。但**微服务拆分后顺序就很关键了** —— 跨服务不能用同一个事务，得用 Seata 的 TCC 模式来保证。

## 三、订单详情（OrderDetail）

`buildDetails` 方法把商品信息"快照"到订单详情表：

```java
private List<OrderDetail> buildDetails(Long orderId, List<ItemDTO> items, ...) {
    for (ItemDTO item : items) {
        OrderDetail detail = new OrderDetail();
        detail.setName(item.getName());      // 商品名
        detail.setSpec(item.getSpec());       // 规格
        detail.setPrice(item.getPrice());     // 下单时的价格
        detail.setNum(numMap.get(item.getId())); // 数量
        detail.setItemId(item.getItemId());   // 商品ID
        detail.setImage(item.getImage());     // 图片
        detail.setOrderId(orderId);           // 关联订单
    }
}
```

**为什么要冗余一份商品信息？** 因为商品价格会变。用户 1 月下单时价格是 100，2 月查看订单时商品已涨价到 150，但订单详情里记录的还是 100。**订单详情是交易快照，不受商品后续变更影响。**

## 四、标记支付成功（markOrderPaySuccess）

```java
public void markOrderPaySuccess(Long orderId) {
    Order order = new Order();
    order.setId(orderId);
    order.setStatus(2);  // 2=已付款
    order.setPayTime(LocalDateTime.now());
    updateById(order);
}
```

- 由支付模块在支付成功后调用
- 只改状态和支付时间，逻辑很简单

## 五、订单状态流转

```
创建订单(status=1)
    ↓
支付成功(status=2)
    ↓
发货(status=3)
    ↓
确认收货(status=4)
    ↓
完成(status=5)

特殊状态：
  status=6 → 已关闭（超时未支付/用户取消）
  status=7 → 已评价
```

## 六、踩坑避坑指南

### 坑 1：总价在应用层算而不是数据库

- 问题：`total += item.getPrice() * itemNumMap.get(item.getId())`
- 风险：如果商品查出来是旧价格（缓存），算出的总价可能不对
- 当前方案没问题（实时查数据库），但微服务拆分后要注意

### 坑 2：商品不存在时的校验不够严格

```java
if (items == null || items.size() < itemIds.size()) {
    throw new BadRequestException("商品不存在");
}
```

- 只检查了数量是否匹配，没检查具体哪个商品不存在
- 用户体验不好，应该返回具体哪个商品不存在

### 坑 3：库存不足时购物车已经清了

```java
// 6. 清理购物车
cartService.removeByItemIds(itemIds);
// 7. 扣减库存
itemService.deductStock(detailDTOS);  // ← 这里失败会回滚
```

- 单体架构下没问题（事务回滚会恢复购物车）
- 微服务拆分后，购物车清理和库存扣减在不同服务，回滚管不到

## 七、微服务拆分后的问题

| 问题 | 严重程度 | 解决方案 |
|---|---|---|
| 查商品变成远程调用 | ⭐⭐ | OpenFeign |
| 写订单和扣库存不在同一个事务 | ⭐⭐⭐⭐ | Seata 分布式事务 |
| 清购物车和写订单不在同一个事务 | ⭐⭐ | 消息队列最终一致性 |
| 总价计算依赖商品服务 | ⭐⭐⭐ | 要保证价格一致性 |

**核心矛盾：** `@Transactional` 只能管同一个数据库的操作，拆分后四个操作在四个不同数据库里，需要 Seata 来兜底。
