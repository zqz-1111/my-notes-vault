# 【购物车】购物车模块业务流程

## 一、写在前面

1. 购物车是用户和订单之间的"缓冲区"，用户先把想买的放进去，最后一起结算
2. 核心设计：购物车表只存 userId + itemId + num，商品信息实时从商品服务查

## 二、加入购物车（addItem2Cart）

```java
public void addItem2Cart(CartFormDTO cartFormDTO) {
    Long userId = UserContext.getUser();

    // 1. 判断商品是否已在购物车
    if (checkItemExists(cartFormDTO.getItemId(), userId)) {
        // 存在 → 数量 +1
        baseMapper.updateNum(cartFormDTO.getItemId(), userId);
        return;
    }
    // 2. 不存在 → 检查购物车是否满了（上限 10 个）
    checkCartsFull(userId);
    // 3. 新增购物车条目
    Cart cart = BeanUtils.copyBean(cartFormDTO, Cart.class);
    cart.setUserId(userId);
    save(cart);
}
```

### 流程图

```
加入购物车
  ├── 商品已存在？
  │     ├── 是 → 数量 +1（updateNum）
  │     └── 否 → 购物车满了？
  │                 ├── 是 → 抛异常"购物车不能超过10个"
  │                 └── 否 → 新增一条记录
```

### 关键细节

- **重复商品不新建记录**：同一个商品加两次，只改数量，不创建两条
- **购物车有上限**：最多 10 个不同商品，防止恶意刷数据
- **不存商品信息**：只存 `itemId`，价格、库存等信息查询时实时取

## 三、查询购物车（queryMyCarts）

```java
public List<CartVO> queryMyCarts() {
    // 1. 查当前用户的购物车列表
    List<Cart> carts = lambdaQuery().eq(Cart::getUserId, UserContext.getUser()).list();
    // 2. 转 VO
    List<CartVO> vos = BeanUtils.copyList(carts, CartVO.class);
    // 3. 填充商品信息
    handleCartItems(vos);
    return vos;
}
```

### 填充商品信息（handleCartItems）

```java
private void handleCartItems(List<CartVO> vos) {
    // 1. 收集所有 itemId
    Set<Long> itemIds = vos.stream().map(CartVO::getItemId).collect(Collectors.toSet());
    // 2. 批量查商品（一次请求拿所有商品，不是逐个查）
    List<ItemDTO> items = itemService.queryItemByIds(itemIds);
    // 3. 转成 Map（id → item），方便快速查找
    Map<Long, ItemDTO> itemMap = items.stream()
            .collect(Collectors.toMap(ItemDTO::getId, Function.identity()));
    // 4. 写入 VO
    for (CartVO v : vos) {
        ItemDTO item = itemMap.get(v.getItemId());
        v.setNewPrice(item.getPrice());  // 最新价格
        v.setStatus(item.getStatus());   // 商品状态
        v.setStock(item.getStock());     // 库存
    }
}
```

### 为什么实时查而不冗余存储？

| 方案 | 优点 | 缺点 |
|---|---|---|
| 实时查（当前方案） | 数据永远是最新的，价格改了购物车跟着变 | 多一次查询，依赖商品服务 |
| 冗余存储 | 查询快，不依赖其他服务 | 商品改价后购物车显示旧价格，需要同步机制 |

**单体架构下实时查没问题（同进程调用快），微服务拆分后就变成跨服务调用了，需要权衡。**

## 四、删除购物车（removeByItemIds）

```java
public void removeByItemIds(Collection<Long> itemIds) {
    QueryWrapper<Cart> queryWrapper = new QueryWrapper<>();
    queryWrapper.lambda()
            .eq(Cart::getUserId, UserContext.getUser())  // 只删自己的
            .in(Cart::getItemId, itemIds);                // 按 itemId 批量删
    remove(queryWrapper);
}
```

- 用在下单成功后，把已下单的商品从购物车移除
- WHERE 条件带 userId，保证只能删自己的购物车（权限控制）

## 五、购物车与订单的协作

```
用户点击"去结算"
  ↓
OrderService.createOrder()
  ├── 从购物车拿到选中的商品 itemId 列表
  ├── 调商品服务查价格，算总价
  ├── 写入订单
  └── 调 cartService.removeByItemIds() 清理已下单的商品
```

## 六、踩坑避坑指南

### 坑 1：查询购物车时空指针

- 问题：购物车里有 itemId，但商品已下架或删除
- 后果：`itemMap.get(v.getItemId())` 返回 null，直接访问属性会 NPE
- 当前代码处理：`if (item == null) continue;` 跳过

### 坑 2：加入购物车没加锁

- 问题：并发请求同时加入同一个商品
- 后果：可能插入两条相同 itemId 的记录
- 影响：不严重，查询时正常显示，但数据不干净
- 优化方案：数据库加唯一索引 `(user_id, item_id)`，重复插入时走更新逻辑

## 七、微服务拆分后的问题

| 问题 | 说明 |
|---|---|
| 查商品信息变成远程调用 | handleCartItems 调 itemService 变成 HTTP 请求，N 个商品可能 N 次调用 |
| 下单清购物车跨服务 | 订单服务和购物车服务不在同一个事务里，订单创建成功但购物车清理失败怎么办？ |
| 商品信息更新感知 | 商品服务改了价格，购物车服务不知道，只有下次查询才能拿到最新价格 |
