# 【商品】商品模块业务流程

## 一、写在前面

1. 商品模块是整个商城的基础，其他模块都依赖它获取商品信息、扣减库存
2. 核心难点：库存扣减的并发安全问题

## 二、模块职责

商品模块干两件事：
- **查商品**：根据 ID 查询商品详情，供购物车、订单等模块使用
- **扣库存**：下单时扣减商品库存，保证不超卖

## 三、查商品（queryItemByIds）

```java
@Override
public List<ItemDTO> queryItemByIds(Collection<Long> ids) {
    return BeanUtils.copyList(listByIds(ids), ItemDTO.class);
}
```

- `listByIds` 是 MyBatis-Plus 内置方法，等价于 `SELECT * FROM item WHERE id IN (?,?,?)`
- 返回 ItemDTO（不是 Item PO），用 `BeanUtils.copyList` 做对象转换
- 购物车模块和订单模块都调用这个方法获取商品信息

## 四、扣库存（deductStock）

### 4.1 代码实现

```java
@Override
public void deductStock(List<OrderDetailDTO> items) {
    String sqlStatement = "com.hmall.mapper.ItemMapper.updateStock";
    boolean r = false;
    try {
        r = executeBatch(items, (sqlSession, entity) ->
                sqlSession.update(sqlStatement, entity));
    } catch (Exception e) {
        throw new BizIllegalException("更新库存异常，可能是库存不足!", e);
    }
    if (!r) {
        throw new BizIllegalException("库存不足！");
    }
}
```

### 4.2 SQL 关键（ItemMapper.xml）

```sql
UPDATE item
SET stock = stock - #{num}
WHERE id = #{id} AND stock >= #{num}
```

**核心思路：不在应用层判断库存够不够，直接在 SQL 里用 `stock >= #{num}` 做条件判断。**

- 如果库存充足：`stock = 10, num = 3` → 更新成功，stock 变成 7
- 如果库存不足：`stock = 2, num = 3` → WHERE 条件不满足，影响行数为 0 → 返回 false

### 4.3 为什么不用"先查再扣"？

```
错误方案（有并发问题）：
  线程 A：查库存 = 1 → 够 → 扣减
  线程 B：查库存 = 1 → 够 → 扣减
  结果：库存扣成 -1，超卖了！

正确方案（当前实现）：
  线程 A：UPDATE ... WHERE stock >= 1 → 成功
  线程 B：UPDATE ... WHERE stock >= 1 → 失败（stock 已经是 0）
```

这和 hm-dianping 里秒杀扣库存的 CAS 思路是一样的 —— **把判断和更新合在一个原子操作里**。

### 4.4 批量执行

用的是 MyBatis-Plus 的 `executeBatch`，原理是开启批处理模式后循环执行同一条 SQL：

```java
executeBatch(items, (sqlSession, entity) ->
    sqlSession.update(sqlStatement, entity));
```

底层走的是 JDBC 的 `addBatch()` + `executeBatch()`，比循环单条执行快很多（减少网络往返）。

## 五、商品模块被其他模块的调用关系

```
购物车模块 CartServiceImpl
  ├── addItem2Cart() 不直接调商品模块（只存 itemId）
  └── queryMyCarts() → itemService.queryItemByIds() 获取价格/库存/状态

订单模块 OrderServiceImpl
  ├── createOrder() → itemService.queryItemByIds() 获取商品信息算总价
  └── createOrder() → itemService.deductStock() 扣减库存
```

**关键点：** 商品信息不是冗余存储在购物车/订单表里的，而是实时查询。好处是数据一致（价格改了购物车跟着变），坏处是多了一次调用。微服务拆分后这就是跨服务调用的性能问题。

## 六、踩坑避坑指南

### 坑 1：扣库存用 = 而不是 -=

- 错误写法：`SET stock = #{newStock}` （先查出来再算）
- 正确写法：`SET stock = stock - #{num}`（数据库自己算）
- 原因：并发场景下，两个请求读到的 stock 值可能一样，各自算完写回去会覆盖

### 坑 2：executeBatch 没开批处理

- 问题：MyBatis 的 `executeBatch` 需要在事务环境下才能真正走批处理
- 后果：看起来是批量，实际还是逐条执行
- 正确做法：调用方加 `@Transactional` 注解

## 七、微服务拆分后的问题

| 问题 | 说明 |
|---|---|
| 查商品变成远程调用 | 订单服务不能直接查本地数据库，得通过 HTTP/RPC 调商品服务 |
| 扣库存跨服务 | 订单服务扣库存失败时，前面创建的订单已经写入了 → 需要分布式事务 |
| 性能问题 | 创建订单要先查商品再扣库存，两次远程调用，延迟增加 |
