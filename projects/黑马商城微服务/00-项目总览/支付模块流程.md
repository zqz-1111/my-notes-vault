# 【支付】支付模块业务流程

## 一、写在前面

1. 支付模块是最考验设计功底的部分 —— 涉及金钱操作，任何 bug 都是资损
2. 核心难点：幂等性（重复请求不重复扣款）、并发安全（同一笔订单不能支付两次）
3. 当前项目只实现了余额支付，实际项目还会对接支付宝/微信支付

## 二、支付流程概览

```
创建订单成功
    ↓
申请支付单（applyPayOrder）← 幂等性校验
    ↓
余额支付（tryPayOrderByBalance）
    ├── 扣用户余额
    ├── 修改支付单状态
    └── 修改订单状态
```

## 三、申请支付单（applyPayOrder）

### 3.1 入口

```java
@Override
public String applyPayOrder(PayApplyDTO applyDTO) {
    PayOrder payOrder = checkIdempotent(applyDTO);
    return payOrder.getId().toString();
}
```

### 3.2 幂等性校验（checkIdempotent）

这是整个支付模块最核心的方法，六步判断：

```java
private PayOrder checkIdempotent(PayApplyDTO applyDTO) {
    // 1. 根据业务订单号查支付单
    PayOrder oldOrder = queryByBizOrderNo(applyDTO.getBizOrderNo());

    // 2. 不存在 → 第一次申请，新建支付单
    if (oldOrder == null) {
        PayOrder payOrder = buildPayOrder(applyDTO);
        payOrder.setPayOrderNo(IdWorker.getId());
        save(payOrder);
        return payOrder;
    }

    // 3. 已支付 → 抛异常"订单已经支付"
    if (PayStatus.TRADE_SUCCESS.equalsValue(oldOrder.getStatus())) {
        throw new BizIllegalException("订单已经支付！");
    }

    // 4. 已关闭 → 抛异常"订单已关闭"
    if (PayStatus.TRADE_CLOSED.equalsValue(oldOrder.getStatus())) {
        throw new BizIllegalException("订单已关闭");
    }

    // 5. 支付渠道变了 → 重置旧单数据，重新申请
    if (!StringUtils.equals(oldOrder.getPayChannelCode(), applyDTO.getPayChannelCode())) {
        PayOrder payOrder = buildPayOrder(applyDTO);
        payOrder.setId(oldOrder.getId());  // 复用旧单 ID
        updateById(payOrder);
        return payOrder;
    }

    // 6. 旧单存在、未支付、渠道一致 → 直接返回旧单
    return oldOrder;
}
```

### 3.3 六步判断的流程图

```
申请支付单
  ├── 旧单存在？
  │     └── 否 → 新建支付单，返回
  │
  ├── 旧单已支付？
  │     └── 是 → 抛异常"已支付"
  │
  ├── 旧单已关闭？
  │     └── 是 → 抛异常"已关闭"
  │
  ├── 支付渠道变了？（比如从支付宝换成微信）
  │     └── 是 → 重置旧单数据，返回
  │
  └── 旧单存在 + 未支付 + 渠道一致 → 直接返回旧单
```

**核心思想：同一个订单不管请求多少次，只会有一张支付单。** 这就是幂等性 —— 相同的请求执行多次，结果和执行一次一样。

### 3.4 为什么需要幂等性？

实际场景：
- 用户网络卡了，点了两次"支付"按钮
- 前端重试机制自动重发请求
- 支付回调通知可能重复发送

如果没有幂等性 → 创建两张支付单 → 扣两次钱 → 资损。

## 四、余额支付（tryPayOrderByBalance）

```java
@Transactional
public void tryPayOrderByBalance(PayOrderFormDTO payOrderFormDTO) {
    // 1. 查询支付单
    PayOrder po = getById(payOrderFormDTO.getId());

    // 2. 判断状态必须是"待支付"
    if (!PayStatus.WAIT_BUYER_PAY.equalsValue(po.getStatus())) {
        throw new BizIllegalException("交易已支付或关闭！");
    }

    // 3. 扣减余额（调用户模块）
    userService.deductMoney(payOrderFormDTO.getPw(), po.getAmount());

    // 4. 修改支付单状态（乐观锁）
    boolean success = markPayOrderSuccess(payOrderFormDTO.getId(), LocalDateTime.now());
    if (!success) {
        throw new BizIllegalException("交易已支付或关闭！");
    }

    // 5. 修改订单状态为已付款
    Order order = new Order();
    order.setId(po.getBizOrderNo());
    order.setStatus(2);
    order.setPayTime(LocalDateTime.now());
    orderService.updateById(order);
}
```

### 4.1 五步流程

```
余额支付
  ├── 查支付单
  ├── 状态是"待支付"？不是 → 抛异常
  ├── 扣用户余额（userService.deductMoney）
  ├── 支付单状态 → 成功（乐观锁）
  └── 订单状态 → 已付款
```

### 4.2 乐观锁防并发（markPayOrderSuccess）

```java
public boolean markPayOrderSuccess(Long id, LocalDateTime successTime) {
    return lambdaUpdate()
            .set(PayOrder::getStatus, PayStatus.TRADE_SUCCESS.getValue())
            .set(PayOrder::getPaySuccessTime, successTime)
            .eq(PayOrder::getId, id)
            // 乐观锁：只有状态是"未提交"或"待支付"才能改成"成功"
            .in(PayOrder::getStatus,
                PayStatus.NOT_COMMIT.getValue(),
                PayStatus.WAIT_BUYER_PAY.getValue())
            .update();
}
```

**为什么需要乐观锁？**

```
并发场景（没有乐观锁）：
  线程 A：读状态 = 待支付 → 扣余额成功 → 更新状态为成功
  线程 B：读状态 = 待支付 → 扣余额成功 → 更新状态为成功
  结果：扣了两次余额！

有乐观锁：
  线程 A：UPDATE ... WHERE status IN (待支付, 未提交) → 成功，状态变成成功
  线程 B：UPDATE ... WHERE status IN (待支付, 未提交) → 失败（状态已经是成功了）
  结果：只扣一次
```

### 4.3 扣余额逻辑（UserServiceImpl.deductMoney）

```java
public void deductMoney(String pw, Integer totalFee) {
    // 1. 校验支付密码
    User user = getById(UserContext.getUser());
    if (user == null || !passwordEncoder.matches(pw, user.getPassword())) {
        throw new BizIllegalException("用户密码错误");
    }
    // 2. 扣减余额（SQL 层面保证余额不会扣成负数）
    baseMapper.updateMoney(UserContext.getUser(), totalFee);
}
```

扣余额和扣库存思路一样 —— SQL 里用 `money >= #{amount}` 做条件判断，不在应用层判断。

## 五、支付单与订单分离设计

两张表，各管各的：

| 表 | 职责 | 说明 |
|---|---|---|
| `order` | 业务订单 | 记录买了什么、总价、状态 |
| `pay_order` | 支付单 | 记录支付尝试、支付渠道、支付状态 |

**为什么不直接在订单表里加支付字段？**

- 一个订单可能换支付方式：先试支付宝超时了，换微信支付
- 每次支付尝试都是一条支付单记录
- 支付单有独立的状态流转（未提交 → 待支付 → 支付成功/关闭）
- 订单只关心最终结果（付了没），不关心过程

## 六、支付单状态流转

```
NOT_COMMIT（未提交）
    ↓
WAIT_BUYER_PAY（待支付）← 用户发起支付
    ↓
TRADE_SUCCESS（支付成功）← 余额扣减成功
    或
TRADE_CLOSED（交易关闭）← 超时未支付 / 用户取消
```

## 七、踩坑避坑指南

### 坑 1：支付不加幂等

- 问题：同一个订单重复请求创建多个支付单
- 后果：用户被扣多次钱
- 正确做法：`checkIdempotent()` 先查旧单再决定

### 坑 2：乐观锁条件写错

- 错误：`.eq(PayOrder::getStatus, PayStatus.WAIT_BUYER_PAY.getValue())`
- 后果：NOT_COMMIT 状态的单子改不了
- 正确：`.in(PayOrder::getStatus, NOT_COMMIT, WAIT_BUYER_PAY)`

### 坑 3：先扣余额再改状态，中间挂了

```java
userService.deductMoney(...)           // ← 余额扣了
boolean success = markPayOrderSuccess  // ← 这里挂了，状态没改
```

- 问题：余额扣了但支付单状态没变，用户看到"未支付"但钱已经没了
- 单体架构下 `@Transactional` 会回滚余额扣减
- 微服务拆分后，余额扣减和状态更新在不同服务 → 需要分布式事务或补偿机制

### 坑 4：支付密码和登录密码混用

- 当前实现：`deductMoney` 里校验的是用户的支付密码
- 但 `User` 表只有一个 `password` 字段，实际上登录密码和支付密码是同一个
- 实际项目中应该分开：登录密码 + 支付密码（两套）

## 八、微服务拆分后的问题

| 问题 | 严重程度 | 说明 |
|---|---|---|
| 扣余额跨服务 | ⭐⭐⭐⭐ | 支付服务调用户服务扣余额，失败了要回滚 |
| 支付单和订单状态一致性 | ⭐⭐⭐ | 支付单改成功了但订单状态没改 → 数据不一致 |
| 重复支付 | ⭐⭐⭐⭐⭐ | 并发场景下乐观锁在分布式环境可能失效 → 需要分布式锁 |
