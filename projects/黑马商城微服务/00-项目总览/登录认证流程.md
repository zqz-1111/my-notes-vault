# 【认证】单体架构下的登录与鉴权全流程

## 一、写在前面

1. 解决什么问题：用户登录后，服务端怎么识别"这个请求是谁发的"？
2. 学完你能掌握：JWT 认证机制、拦截器鉴权、ThreadLocal 用户上下文的完整链路

## 二、整体架构

```
前端登录 → 后端校验密码 → 生成 JWT Token → 返回给前端
前端每次请求 → 带上 Token → 拦截器校验 → 存入 ThreadLocal → 业务代码取用
```

## 三、登录阶段（获取 Token）

### 3.1 入口

```java
// UserController.java
@PostMapping("login")
public UserLoginVO login(@RequestBody @Validated LoginFormDTO loginFormDTO){
    return userService.login(loginFormDTO);
}
```

### 3.2 业务逻辑（UserServiceImpl.login）

完整流程：

```java
public UserLoginVO login(LoginFormDTO loginDTO) {
    // 1. 根据用户名查数据库
    User user = lambdaQuery().eq(User::getUsername, username).one();
    Assert.notNull(user, "用户名错误");

    // 2. 判断用户是否被冻结
    if (user.getStatus() == UserStatus.FROZEN) {
        throw new ForbiddenException("用户被冻结");
    }

    // 3. 校验密码（BCrypt 加密比对，不是明文比较）
    if (!passwordEncoder.matches(password, user.getPassword())) {
        throw new BadRequestException("用户名或密码错误");
    }

    // 4. 生成 JWT Token
    String token = jwtTool.createToken(user.getId(), jwtProperties.getTokenTTL());

    // 5. 封装 VO 返回（userId + username + balance + token）
    UserLoginVO vo = new UserLoginVO();
    vo.setUserId(user.getId());
    vo.setUsername(user.getUsername());
    vo.setBalance(user.getBalance());
    vo.setToken(token);
    return vo;
}
```

### 3.3 Token 生成原理（JwtTool.createToken）

```java
public String createToken(Long userId, Duration ttl) {
    return JWT.create()
            .setPayload("user", userId)        // 把 userId 塞进 payload
            .setExpiresAt(new Date(...))         // 设置过期时间
            .setSigner(jwtSigner)                // RS256 私钥签名
            .sign();
}
```

- 使用 **RS256 非对称加密**：私钥签名，公钥验证
- 私钥存在 `hmall.jks` 密钥库里，只有服务端有
- 客户端拿到的 Token 里有 userId，但无法伪造（没有私钥）

## 四、鉴权阶段（校验 Token）

### 4.1 拦截器注册（MvcConfig）

```java
@Override
public void addInterceptors(InterceptorRegistry registry) {
    LoginInterceptor loginInterceptor = new LoginInterceptor(jwtTool);
    InterceptorRegistration registration = registry.addInterceptor(loginInterceptor);

    // 配置拦截路径（从 application.yaml 读取）
    registration.addPathPatterns(authProperties.getIncludePaths());
    // 配置放行路径
    registration.excludePathPatterns(authProperties.getExcludePaths());
    // Swagger 路径始终放行
    registration.excludePathPatterns("/error", "/favicon.ico",
            "/v2/**", "/v3/**", "/swagger-resources/**", "/webjars/**", "/doc.html");
}
```

### 4.2 拦截器执行（LoginInterceptor）

```java
@Override
public boolean preHandle(HttpServletRequest request, ...) {
    // 1. 从请求头取 token
    String token = request.getHeader("authorization");
    // 2. 解析校验 token，拿到 userId
    Long userId = jwtTool.parseToken(token);
    // 3. 存入 ThreadLocal
    UserContext.setUser(userId);
    // 4. 放行
    return true;
}

@Override
public void afterCompletion(...) {
    // 请求结束后清理 ThreadLocal（防内存泄漏 + 数据污染）
    UserContext.removeUser();
}
```

### 4.3 Token 解析校验（JwtTool.parseToken）

五步校验，任何一步失败都抛 UnauthorizedException：

```java
public Long parseToken(String token) {
    // 1. token 为空 → "未登录"
    // 2. 签名验证失败 → "无效的token"
    // 3. token 过期 → "token已经过期"
    // 4. payload 中 user 字段为空 → "无效的token"
    // 5. userId 格式错误 → "无效的token"
    // 全部通过 → 返回 userId
}
```

### 4.4 UserContext（ThreadLocal 工具类）

```java
public class UserContext {
    private static final ThreadLocal<Long> tl = new ThreadLocal<>();

    public static void setUser(Long userId) { tl.set(userId); }
    public static Long getUser() { return tl.get(); }
    public static void removeUser() { tl.remove(); }
}
```

业务代码里直接 `UserContext.getUser()` 就能拿到当前登录用户的 ID，不用每个方法都传参。

## 五、ThreadLocal 生命周期

**核心：ThreadLocal 的生命周期 = 一次 HTTP 请求的生命周期。**

```
请求进入 → preHandle 存 userId
    ↓
Controller / Service 使用 UserContext.getUser()
    ↓
afterCompletion 清理 userId
```

### 为什么必须清理？

Tomcat 用线程池复用线程，不清理的话：

```
请求 A（张三）→ 线程 1 → 存了张三 ID → 没清理
请求 B（李四）→ 复用线程 1 → 拿到的是张三 ID！→ 李四操作扣了张三的钱
```

### 如果中途抛异常呢？

`afterCompletion` 不管有没有异常都会执行，所以 ThreadLocal 一定会被清理。

```
│ 前端                                                 │

│ 每次请求 Header 加 Authorization: <token>             │

```

## 七、踩坑避坑指南

### 坑 1：ThreadLocal 不清理导致脏数据

- 问题：忘记在 afterCompletion 里 removeUser()
- 后果：线程池复用时，下一个请求读到上一个用户的数据
- 正确做法：afterCompletion 里必须 removeUser()

### 坑 2：密码用 == 比较

- 问题：`user.getPassword().equals(password)` 明文比对
- 后果：数据库存的是 BCrypt 密文，永远匹配不上
- 正确做法：用 `passwordEncoder.matches(明文, 密文)`

### 坑 3：拦截器顺序问题

- 问题：如果有多个拦截器，LoginInterceptor 放在最后
- 后果：其他拦截器可能拿不到 UserContext 里的用户信息
- 正确做法：认证拦截器要第一个注册

## 八、微服务拆分后的问题

当前方案在单体架构下没问题，但拆分后：

| 问题 | 说明 |
|---|---|
| ThreadLocal 不跨服务 | 请求到服务 A 后，A 调 B 时 ThreadLocal 传不过去 |
| 每个服务都要校验 Token | 用户请求可能直接打到任意服务，不能只在网关校验 |
| Token 里的信息不够 | 微服务可能需要更多用户信息（角色、权限），不只一个 userId |

→ 后续解决方案：Gateway 统一鉴权 + 服务间传递用户信息
