# 【Sentinel】服务保护方案理论入门

## 一、写在前面

微服务架构下，服务之间相互调用。一旦某个服务出问题，会像多米诺骨牌一样拖垮整个集群。今天学的就是怎么防止这种**雪崩问题**。

学完这篇你能知道：
1. 雪崩问题是怎么产生的
2. 四种服务保护方案的原理和分工
3. Sentinel 和 Hystrix 怎么选

## 二、雪崩问题

### 2.1 什么是雪崩

微服务相互调用时，一个服务故障 → 调用方也被拖垮 → 级联失败 → 整个集群不可用。

### 2.2 产生原因（三步演进）

```
第一步：服务提供者故障或阻塞（根因）
    ↓
第二步：调用者没有做好异常处理，自己也被拖垮（传导）
    ↓
第三步：所有有调用关系的服务级联失败（雪崩）
```

举个例子：商品服务并发高 → 响应变慢 → 购物车服务调商品服务也要等 → 购物车服务 Tomcat 线程被占满 → 购物车所有接口都慢 → 调用购物车的服务也跟着慢 → 整个集群瘫痪。

### 2.3 解决思路

| 思路 | 说明 |
|---|---|
| 尽量避免服务故障或阻塞 | 保证代码健壮性、网络畅通、应对高并发 |
| 做好后备方案 | 调用失败时返回默认数据或友好提示，不让异常扩散 |

## 三、四大服务保护方案

### 3.1 请求限流

**问题：** 服务故障最重要的原因是并发太高。

**方案：** 限制或控制接口访问的并发流量，避免服务因流量激增而故障。

**类比：** 水电站大坝，蓄水控制流出大小，让下游水流始终平稳。

### 3.2 线程隔离

**问题：** 一个接口响应慢 + 并发高 → 耗尽 Tomcat 线程 → 其他接口也受影响。

**方案：** 限定每个接口可以使用的线程数量上限，将其"隔离"起来。

**类比：** 轮船舱壁模式 —— 船舱被隔板分割为 N 个密闭舱，触礁进水只影响损坏的部分，不会整船沉没。

**例子：** 查询购物车限定最多 20 个线程，即使商品服务故障导致这 20 个线程阻塞，也不会影响其他接口。

### 3.3 服务熔断

**问题：** 线程隔离避免了雪崩，但故障服务依然会拖慢调用方，购物车查询依然会失败。

**方案：** 两件事：
1. **编写降级逻辑：** 调用失败后的处理 —— 抛异常、返回友好提示、返回默认数据
2. **异常统计和熔断：** 统计接口异常比例，过高则拒绝调用，直接走降级逻辑

### 3.4 失败处理（Fallback）

**方案：** 定义 fallback 逻辑，业务失败时不抛异常，返回默认数据或友好提示。

### 3.5 四者关系

```
请求进来
  ↓
┌─────────────┐
│  请求限流    │ ← 第一道关：控制总量，防过载
└─────┬───────┘
      ↓
┌─────────────┐
│  线程隔离    │ ← 第二道关：某个接口挂了，不影响其他
└─────┬───────┘
      ↓
┌─────────────┐
│  服务熔断    │ ← 第三道关：对方挂了，不再调用，走降级
└─────┬───────┘
      ↓
   正常响应 / 降级响应
```

都是**服务降级**的方案 —— 牺牲部分体验，换取整体健壮性。

## 四、技术选型：Sentinel vs Hystrix

| 维度 | Sentinel ✅ | Hystrix ❌ |
|---|---|---|
| 线程隔离 | 信号量隔离（轻量） | 线程池隔离（开销大）/ 信号量 |
| 熔断策略 | 慢调用比例 + 异常比例 | 仅异常比率 |
| 限流 | 基于 QPS，支持流量整形 | 有限支持 |
| Fallback | ✅ | ✅ |
| 控制台 | 开箱即用，监控+规则配置 | 不完善 |
| 配置方式 | 控制台配置，重启失效 | 注解/配置文件，永久生效 |

**选择 Sentinel：**
- 功能更全（限流+隔离+熔断+监控一体化）
- 阿里开源，Spring Cloud Alibaba 原生支持
- 控制台体验好
- Hystrix 已停止维护

> Sentinel 配置重启失效的问题，后续通过 Nacos 持久化规则解决。
