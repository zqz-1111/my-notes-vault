# 延迟消息实战：订单超时自动取消

## 一、写在前面

1. 为什么学这个？—— 用户下单后不付款，订单不能一直占着库存，需要超时自动关闭
2. 学完你能掌握什么 —— 用 DelayExchange 插件实现延迟消息、完整链路：下单→延迟→查状态→关单→恢复库存

---

## 二、业务需求

用户下单后如果 15 分钟内没支付，系统自动：
1. 关闭订单（状态改为 5）
2. 恢复库存

**之前的做法**：前端显示 30 分钟倒计时，后端没有实际实现

**现在的做法**：后端用 RabbitMQ 延迟消息，精确控制超时时间

---

## 三、整体流程

```
用户下单（createOrder）
    ↓
    ├── 扣库存（同步）
    ├── 清理购物车（异步MQ）
    └── 发延迟消息（orderId, delay=15min）
             ↓ 15分钟后
    OrderDelayMessageListener 消费
             ↓
       查订单状态 → 已支付？→ 不处理
             ↓ 未支付
       查支付流水 → status=3？→ 标记已支付
             ↓ status≠3
       cancelOrder() → 关单 + 恢复库存
```

## 四、代码实现

### 4.1 定义 MQ 常量

```java
// com.hmall.trade.constants.MQConstants
public interface MQConstants {
    String DELAY_EXCHANGE_NAME = "trade.delay.direct";
    String DELAY_ORDER_QUEUE_NAME = "trade.delay.order.queue";
    String DELAY_ORDER_KEY = "delay.order.query";
}
```

**作用**：把交换机名、队列名、路由键抽成常量，避免硬编码魔法值散落在各处。

### 4.2 下单时发送延迟消息

```java
// OrderServiceImpl.createOrder()
// 5. 发送延迟消息，检测订单支付状态
rabbitMqHelper.sendDelayMessage(
    MQConstants.DELAY_EXCHANGE_NAME,
    MQConstants.DELAY_ORDER_KEY,
    order.getId(),                    // 消息体：订单ID
    15 * 60 * 1000                    // 延迟15分钟
);
```

**注意**：消息体只传订单 ID，不需要传整个订单对象，消费者收到后自己查数据库。

### 4.3 消费者监听器

```java
@Component
@RequiredArgsConstructor
public class OrderDelayMessageListener {

    private final IOrderService orderService;
    private final PayClient payClient;

    @RabbitListener(bindings = @QueueBinding(
            value = @Queue(name = MQConstants.DELAY_ORDER_QUEUE_NAME),
            exchange = @Exchange(name = MQConstants.DELAY_EXCHANGE_NAME, delayed = "true"),
            key = MQConstants.DELAY_ORDER_KEY
    ))
    public void listenOrderDelayMessage(Long orderId){
        // 1.查询订单
        Order order = orderService.getById(orderId);
        // 2.已支付或不存在 → 不处理
        if(order == null || order.getStatus() != 1){
            return;
        }
        // 3.查支付流水
        PayOrderDTO payOrder = payClient.queryPayOrderByBizOrderNo(orderId);
        // 4.判断结果
        if(payOrder != null && payOrder.getStatus() == 3){
            // 漏网之鱼：实际已支付但状态没更新
            orderService.markOrderPaySuccess(orderId);
        }else{
            // 真没付 → 关单
            orderService.cancelOrder(orderId);
        }
    }
}
```

**为什么要查支付流水？** 防极端情况：用户在第 14 分 59 秒支付成功，但消息队列处理延迟，订单状态还没来得及更新。查一下支付流水，发现 status=3，就补标为已支付，而不是误关订单。

### 4.4 cancelOrder 实现

```java
@Override
public void cancelOrder(Long orderId) {
    // 1.查询订单
    Order order = getById(orderId);
    if (order == null || order.getStatus() != 1) {
        return;  // 幂等性：已取消或已支付的不重复处理
    }
    // 2.更新订单状态为取消
    Order update = new Order();
    update.setId(orderId);
    update.setStatus(5);           // 5=交易取消
    update.setCloseTime(LocalDateTime.now());
    updateById(update);

    // 3.查询订单详情，恢复库存
    List<OrderDetail> details = detailService.lambdaQuery()
        .eq(OrderDetail::getOrderId, orderId)
        .list();
    List<OrderDetailDTO> detailDTOS = BeanUtils.copyList(details, OrderDetailDTO.class);
    itemClient.restoreStock(detailDTOS);
}
```

## 五、幂等性设计

```java
if (order == null || order.getStatus() != 1) {
    return;  // status 已经是 5 了，不是 1，直接退出
}
```

**幂等** = 同一个操作执行多次，结果跟执行一次一样。

**为什么需要？** 延迟消息不保证只消费一次，可能的情况：
1. 网络抖动：消费者处理完了但 ACK 丢了，RabbitMQ 重新投递
2. 消费者重启：处理到一半挂了，消息回到队列重新消费

**怎么做的？** 只有 `status = 1`（未支付）的订单才处理，已经是 5（取消）或 2（已支付）的直接 return，不会重复恢复库存。
