# 初识MQ：同步调用问题与异步架构

## 一、写在前面

在微服务架构中，服务间通信是核心问题。我们之前基于 OpenFeign 的同步调用方式，在业务复杂度上升后会暴露出严重问题。MQ（消息队列）就是为了解决这些问题而生的。

**学完这篇你能掌握：**
- 同步调用的三大痛点（拓展性差、性能下降、级联失败）
- 异步调用的核心模型与优势
- 主流 MQ 技术选型对比

## 二、同步调用的问题

### 2.1 以余额支付为例

目前采用 OpenFeign 同步调用，支付流程：

```
支付服务 ──同步──→ 用户服务（扣余额）    50ms
    │
    ├──同步──→ 更新支付流水单状态        50ms
    │
    ├──同步──→ 交易服务（更新订单状态）  50ms
    │
    └──同步──→ 通知服务（发短信）        50ms

总耗时 = 50 + 50 + 50 + 50 = 200ms（串行阻塞）
```

### 2.2 三大痛点

#### 痛点一：拓展性差

每次有新需求，支付逻辑都要改：
- 加短信通知 → 改支付代码
- 加积分奖励 → 改支付代码
- 加金币返还 → 改支付代码

**违反开闭原则**，代码越来越臃肿。

#### 痛点二：性能下降

同步调用串行阻塞，总耗时 = 各调用时长之和。每个服务 50ms，4 个调用就是 200ms，用户体验差。

#### 痛点三：级联失败

核心矛盾：**扣款成功了，不能因为短信发不出去就回滚整个事务。**

```
扣款成功 → 更新流水成功 → 更新订单成功 → 发短信失败
                                            ↓
                                    整个事务回滚！
                                            ↓
                                    钱扣了但支付失败？❌
```

**核心业务**（扣款、更新流水）和**非核心业务**（发短信、积分）绑在一起，非核心的失败会拖垮核心的。

## 三、异步调用模型

### 3.1 三个角色

```
┌─────────────┐     消息      ┌─────────────┐     分发      ┌─────────────┐
│  消息发送者   │ ──────────→ │  消息Broker  │ ──────────→ │  消息接收者   │
│ （支付服务）  │             │ （消息队列）  │             │ （交易/通知） │
└─────────────┘             └─────────────┘             └─────────────┘
```

- **消息发送者**：投递消息的人，原来的调用方
- **消息 Broker**：管理、暂存、转发消息，类似微信服务器
- **消息接收者**：接收和处理消息的人，原来的服务提供方

### 3.2 支付业务改造

```
支付服务 ──同步──→ 用户服务（扣余额）
    │
    ├──同步──→ 更新支付流水单
    │
    └──发MQ消息──→ [Broker]
                      │
                      ├──→ 交易服务（更新订单）
                      ├──→ 通知服务（发短信）
                      ├──→ 积分服务（返积分）  ← 新增，支付代码不动
                      └──→ 金币服务（返金币）  ← 新增，支付代码不动
```

核心业务同步保证一致性，非核心业务异步解耦。

### 3.3 异步调用的优势

| 优势 | 说明 |
|---|---|
| **耦合度更低** | 通过 Broker 中转，发送者和接收者完全解耦 |
| **性能更好** | 只等核心业务，非核心异步处理，耗时从 200ms 降到 100ms |
| **业务拓展性强** | 新服务订阅消息即可，支付代码完全不用改 |
| **故障隔离** | 通知服务挂了不影响支付，避免级联失败 |

### 3.4 异步调用的代价

| 缺点 | 说明 |
|---|---|
| **依赖 Broker** | Broker 挂了 → 整个异步链路全断，消息丢失 |
| **架构复杂** | 消息丢失、重复消费、顺序性、幂等性都要自己处理 |
| **调试困难** | 同步调用栈清晰，异步排查问题要翻日志 + 追踪消息链路 |

**一句话总结：** 异步调用用**架构复杂度**换**业务解耦 + 性能 + 扩展性**。Broker 的可靠性成了整个系统的关键。

## 四、MQ 技术选型

### 4.1 主流 MQ 对比

| 维度 | RabbitMQ | RocketMQ | Kafka | ActiveMQ |
|---|---|---|---|---|
| **开发语言** | Erlang | Java | Scala/Java | Java |
| **可用性** | 高 | 高 | 非常高 | 中 |
| **可靠性** | 高 | 高 | 高 | 中 |
| **吞吐量** | 中 | 高 | 非常高 | 低 |
| **延迟** | 微秒级 | 毫秒级 | 毫秒级 | 毫秒级 |
| **路由灵活性** | 非常灵活 | 一般 | 一般 | 灵活 |
| **Spring 生态** | 完善 | 完善 | 完善 | 一般 |

### 4.2 选型决策树

```
需要事务消息 / 电商场景？ → RocketMQ
日志采集 / 大数据？      → Kafka
中小规模 / 复杂路由？    → RabbitMQ
```

### 4.3 各维度推荐

| 维度 | 推荐 | 原因 |
|---|---|---|
| **追求可用性** | Kafka、RocketMQ、RabbitMQ | 集群能力强 |
| **追求可靠性** | RabbitMQ、RocketMQ | 消息确认机制完善 |
| **追求吞吐量** | RocketMQ、Kafka | 百万级 QPS |
| **追求低延迟** | RabbitMQ、Kafka | 微秒~毫秒级 |

### 4.4 为什么选 RabbitMQ

- 各方面比较均衡，稳定性好
- 路由灵活（4种交换机类型）
- Spring 生态整合好（SpringAMQP）
- 国内使用量最多，社区活跃
- 学习性价比最高

## 五、全文总结

| 核心知识点 | 要点 |
|---|---|
| 同步调用三大问题 | 拓展性差、性能下降、级联失败 |
| 异步调用核心模型 | 发送者 → Broker → 接收者 |
| 异步调用优势 | 解耦、性能好、扩展性强、故障隔离 |
| 异步调用代价 | 依赖 Broker、架构复杂、调试困难 |
| MQ 选型 | RabbitMQ 均衡全能，中小规模首选 |
