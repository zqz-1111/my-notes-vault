# 交换机实战、声明方式与消息转换器

## 一、写在前面

1. 为什么学这个：上一篇掌握了 SimpleQueue 和 WorkQueues，但实际开发中消息分发远比"一个队列一个人收"复杂。交换机决定了消息怎么路由，声明方式决定了运维成本，消息转换器决定了数据格式。
2. 学完这篇你能掌握：三种交换机的代码实战、注解式声明队列交换机、JSON 消息转换器配置。

## 二、Fanout 交换机（广播）

### 2.1 特点

消息广播到所有绑定的队列，每个队列的消费者都能收到一份。RoutingKey 留空即可。

### 2.2 生产者

```java
@Test
public void testFanoutExchange() {
    String exchangeName = "hmall.fanout";
    String message = "hello, everyone!";
    // Fanout 不需要 RoutingKey，传空字符串
    rabbitTemplate.convertAndSend(exchangeName, "", message);
}
```

### 2.3 消费者

```java
@RabbitListener(queues = "fanout.queue1")
public void listenFanoutQueue1(String msg) {
    System.out.println("消费者1接收到Fanout消息：【" + msg + "】");
}

@RabbitListener(queues = "fanout.queue2")
public void listenFanoutQueue2(String msg) {
    System.out.println("消费者2接收到Fanout消息：【" + msg + "】");
}
```

> 两个消费者都收到同一条消息，广播效果。

## 三、Direct 交换机（路由）

### 3.1 特点

根据 RoutingKey 精确匹配，只有队列的 BindingKey 与消息的 RoutingKey 完全一致，才会收到消息。

### 3.2 生产者

```java
@Test
public void testSendDirectExchangeRed() {
    String exchangeName = "hmall.direct";
    String message = "红色警报！日本乱排核废水，导致海洋生物变异，惊现哥斯拉！";
    rabbitTemplate.convertAndSend(exchangeName, "red", message);
}

@Test
public void testSendDirectExchangeBlue() {
    String exchangeName = "hmall.direct";
    String message = "最新报道，哥斯拉是居民自治巨型气球，虚惊一场！";
    rabbitTemplate.convertAndSend(exchangeName, "blue", message);
}
```

### 3.3 消费者

```java
@RabbitListener(queues = "direct.queue1")
public void listenDirectQueue1(String msg) {
    System.out.println("消费者1接收到direct.queue1的消息：【" + msg + "】");
}

@RabbitListener(queues = "direct.queue2")
public void listenDirectQueue2(String msg) {
    System.out.println("消费者2接收到direct.queue2的消息：【" + msg + "】");
}
```

### 3.4 绑定关系（控制台配置）

| 队列 | BindingKey |
|---|---|
| direct.queue1 | red、blue |
| direct.queue2 | red、yellow |

- 发 `red` → 两个队列都收到
- 发 `blue` → 只有 queue1 收到
- 发 `yellow` → 只有 queue2 收到

## 四、Topic 交换机（通配符）

### 4.1 特点

类似 Direct，但 BindingKey 支持通配符：
- `#`：匹配零个或多个词
- `*`：匹配恰好一个词

### 4.2 通配符示例

| BindingKey | 能匹配 | 不能匹配 |
|---|---|---|
| `china.#` | china.news、china.weather、china.news.sports | japan.news |
| `#.news` | china.news、japan.news、china.sports.news | china.weather |
| `*.*` | china.news、japan.weather | china.news.sports |

### 4.3 生产者

```java
@Test
public void testSendTopicExchange() {
    String exchangeName = "hmall.topic";
    String message = "喜报！孙悟空大战哥斯拉，胜!";
    rabbitTemplate.convertAndSend(exchangeName, "china.news", message);
}
```

### 4.4 消费者

```java
@RabbitListener(queues = "topic.queue1")
public void listenTopicQueue1(String msg) {
    System.out.println("消费者1接收到topic.queue1的消息：【" + msg + "】");
}

@RabbitListener(queues = "topic.queue2")
public void listenTopicQueue2(String msg) {
    System.out.println("消费者2接收到topic.queue2的消息：【" + msg + "】");
}
```

### 4.5 绑定关系

- `topic.queue1`：绑定 `china.#` → 匹配所有以 `china.` 开头的 key
- `topic.queue2`：绑定 `#.news` → 匹配所有以 `.news` 结尾的 key

`china.news` 同时匹配两个通配符，所以两个消费者都收到。

## 五、三种交换机对比

| 类型 | 路由规则 | 适用场景 |
|---|---|---|
| Fanout | 广播到所有绑定队列 | 通知、广播 |
| Direct | RoutingKey 精确匹配 | 精确路由 |
| Topic | RoutingKey 通配符匹配 | 灵活路由 |

## 六、声明队列和交换机

### 6.1 为什么用代码声明

手动在控制台创建容易出错，项目上线交给运维时需要一一对应。代码声明可以自动化，启动时自动创建。

### 6.2 方式一：@Bean 声明（Fanout 示例）

```java
@Configuration
public class FanoutConfig {

    @Bean
    public FanoutExchange fanoutExchange(){
        return new FanoutExchange("hmall.fanout");
    }

    @Bean
    public Queue fanoutQueue1(){
        return new Queue("fanout.queue1");
    }

    @Bean
    public Binding bindingQueue1(Queue fanoutQueue1, FanoutExchange fanoutExchange){
        return BindingBuilder.bind(fanoutQueue1).to(fanoutExchange);
    }

    @Bean
    public Queue fanoutQueue2(){
        return new Queue("fanout.queue2");
    }

    @Bean
    public Binding bindingQueue2(Queue fanoutQueue2, FanoutExchange fanoutExchange){
        return BindingBuilder.bind(fanoutQueue2).to(fanoutExchange);
    }
}
```

### 6.3 方式一：@Bean 声明（Direct 示例）

Direct 模式每个 BindingKey 都要写一个 Binding，比较繁琐：

```java
@Configuration
public class DirectConfig {

    @Bean
    public DirectExchange directExchange(){
        return ExchangeBuilder.directExchange("hmall.direct").build();
    }

    @Bean
    public Queue directQueue1(){
        return new Queue("direct.queue1");
    }

    @Bean
    public Binding bindingQueue1WithRed(Queue directQueue1, DirectExchange directExchange){
        return BindingBuilder.bind(directQueue1).to(directExchange).with("red");
    }

    @Bean
    public Binding bindingQueue1WithBlue(Queue directQueue1, DirectExchange directExchange){
        return BindingBuilder.bind(directQueue1).to(directExchange).with("blue");
    }
    // ... 每个 key 都要写一个 Binding
}
```

### 6.4 方式二：注解声明（推荐）

用 `@QueueBinding` 注解直接在 `@RabbitListener` 上声明，一步到位：

```java
// Direct 模式
@RabbitListener(bindings = @QueueBinding(
        value = @Queue(name = "direct.queue1"),
        exchange = @Exchange(name = "hmall.direct", type = ExchangeTypes.DIRECT),
        key = {"red", "blue"}
))
public void listenDirectQueue1(String msg) {
    System.out.println("消费者1接收到direct.queue1的消息：【" + msg + "】");
}

// Topic 模式
@RabbitListener(bindings = @QueueBinding(
        value = @Queue(name = "topic.queue1"),
        exchange = @Exchange(name = "hmall.topic", type = ExchangeTypes.TOPIC),
        key = "china.#"
))
public void listenTopicQueue1(String msg) {
    System.out.println("消费者1接收到topic.queue1的消息：【" + msg + "】");
}
```

> 注解方式比 @Bean 方式简洁得多，推荐优先使用。

### 6.5 两种声明方式对比

| 方式 | 优点 | 缺点 |
|---|---|---|
| @Bean | 集中管理，一目了然 | 代码量大，Direct 每个 key 要写一个 Binding |
| @QueueBinding 注解 | 简洁，声明和监听合一 | 分散在各个监听方法上 |

## 七、消息转换器

### 7.1 为什么需要

Spring 默认用 JDK 序列化，存在三个问题：
- 数据体积过大
- 有安全漏洞
- 可读性差（控制台看到的是乱码字节）

### 7.2 配置 JSON 转换器

在 publisher 和 consumer 的启动类中都加上：

```java
@Bean
public MessageConverter messageConverter(){
    // 1.定义消息转换器
    Jackson2JsonMessageConverter jackson2JsonMessageConverter = new Jackson2JsonMessageConverter();
    // 2.配置自动创建消息id，用于识别不同消息，也可以在业务中基于ID判断是否是重复消息
    jackson2JsonMessageConverter.setCreateMessageIds(true);
    return jackson2JsonMessageConverter;
}
```

### 7.3 依赖说明

如果项目引入了 `spring-boot-starter-web`，Jackson 已经包含在内，无需额外引入。如果没有（比如本项目的 consumer），需要手动加 `jackson-databind` 依赖：

```xml
<dependency>
    <groupId>com.fasterxml.jackson.core</groupId>
    <artifactId>jackson-databind</artifactId>
</dependency>
```

### 7.4 发送和接收

发送端用 Map（或任意对象），接收端也用 Map 接收，Jackson 自动完成 JSON 序列化/反序列化：

```java
// 发送
@Test
public void testSendMap() {
    Map<String,Object> msg = new HashMap<>();
    msg.put("name", "柳岩");
    msg.put("age", 21);
    rabbitTemplate.convertAndSend("object.queue", msg);
}

// 接收
@RabbitListener(queues = "object.queue")
public void listenSimpleQueueMessage(Map<String, Object> msg) {
    System.out.println("消费者接收到object.queue消息：【" + msg + "】");
}
```

### 7.5 messageId 的作用

配置 `setCreateMessageIds(true)` 后，每条消息会自动生成唯一 ID，便于做幂等性判断（防止重复消费）。

## 八、踩坑避坑指南

### 踩坑1：缺少 Jackson 依赖
- 报错：`java.lang.NoClassDefFoundError: com/fasterxml/jackson/databind/ObjectMapper`
- 原因：consumer 没有引入 `spring-boot-starter-web`，缺少 Jackson
- 解决：在 pom.xml 中添加 `jackson-databind` 依赖

### 踩坑2：交换机和队列未绑定
- 问题：创建了交换机和队列，但忘记绑定
- 后果：消息发送成功但队列收不到
- 解决：在控制台检查 Bindings，或用代码声明自动绑定
