# 死信队列与延迟消息

## 一、写在前面

1. 为什么学这个？—— 订单超时取消、定时任务提醒这类"延迟执行"的需求，用定时轮询太笨，用 MQ 延迟消息才优雅
2. 学完你能掌握什么 —— 死信队列原理、TTL+死信实现延迟消息、DelayExchange 插件方案、两种方案的选型

## 二、死信队列（Dead Letter Queue）

### 什么是死信？

消息变成"死信"（Dead Letter）的三种情况：

| 情况 | 说明 |
|---|---|
| 消息被拒绝 | 消费者调用 `basicReject` 或 `basicNack`，且 `requeue=false` |
| 消息过期 | 消息 TTL 到期，还没被消费 |
| 队列满了 | 队列达到最大长度，新消息挤掉旧消息 |

### 死信交换机（Dead Letter Exchange）

死信不会凭空消失，会被路由到**死信交换机**，再由死信交换机转发到死信队列。

```
普通队列 → 消息变成死信 → 死信交换机 → 死信队列 → 消费者处理
```

#### 声明死信交换机和队列

```java
// 死信交换机
@Bean
public DirectExchange dlxExchange() {
    return new DirectExchange("dlx.direct");
}

// 死信队列
@Bean
public Queue dlxQueue() {
    return new Queue("dlx.queue");
}

// 绑定
@Bean
public Binding dlxBinding() {
    return BindingBuilder.bind(dlxQueue()).to(dlxExchange()).with("dlx");
}
```

#### 普通队列绑定死信交换机

在声明普通队列时，通过 `x-dead-letter-exchange` 参数指定死信交换机：

```java
@Bean
public Queue normalQueue() {
    return QueueBuilder
            .durable("normal.queue")
            .deadLetterExchange("dlx.direct")  // 指定死信交换机
            .deadLetterRoutingKey("dlx")        // 指定死信路由键
            .build();
}
```

## 三、TTL + 死信队列实现延迟消息

### 原理

1. 消息发到普通队列，设置 TTL（比如 30 分钟）
2. 普通队列**没有消费者**，消息静静等待过期
3. 消息过期后变成死信，路由到死信交换机
4. 死信队列有消费者，消费过期消息

```
Publisher → 普通交换机 → 普通队列（TTL=30min，无消费者）
                              ↓ 30分钟后消息过期
                           死信交换机 → 死信队列 → Consumer
```

### 代码实现

#### 设置消息 TTL

```java
@Test
void testTTLMessage() {
    rabbitTemplate.convertAndSend("normal.direct", "normal", "hello", message -> {
        message.getMessageProperties().setExpiration("5000"); // TTL 5秒
        return message;
    });
}
```

#### 设置队列 TTL（统一过期时间）

```java
@Bean
public Queue ttlQueue() {
    return QueueBuilder
            .durable("ttl.queue")
            .ttl(5000)  // 队列中所有消息 5 秒后过期
            .deadLetterExchange("dlx.direct")
            .deadLetterRoutingKey("dlx")
            .build();
}
```

### 踩坑：队首处理不准确

RabbitMQ 的消息过期是**基于追溯方式**实现的，消息在**队首时**才会被检查是否过期。

```
场景：
- 消息 A：TTL = 10 秒
- 消息 B：TTL = 30 秒

正常：A 先过期 → B 再过期 ✅

问题：如果队列堆积，A 还没到队首，B 的 TTL 到了但 A 挡在前面
- B 想过期 → 但 A 还没过期 → B 要等 A 先处理 ❌
```

**结论**：TTL + 死信队列方案，消息过期时间不精确，不适合对时间要求严格的场景。

## 四、DelayExchange 插件（推荐）

### 安装插件

安装 `rabbitmq_delayed_message_exchange` 插件后，直接实现延迟消息，不需要绕道死信队列。

```bash
# Docker 容器内安装
rabbitmq-plugins enable rabbitmq_delayed_message_exchange
```

### 声明延迟交换机

```java
@Bean
public DirectExchange delayExchange() {
    return ExchangeBuilder
            .directExchange("delay.direct")
            .delayed()  // 设置 delay 属性为 true
            .durable(true)
            .build();
}

@Bean
public Queue delayQueue() {
    return new Queue("delay.queue");
}

@Bean
public Binding delayBinding() {
    return BindingBuilder.bind(delayQueue()).to(delayExchange()).with("delay");
}
```

### 发送延迟消息

```java
@Test
void testDelayMessage() {
    rabbitTemplate.convertAndSend("delay.direct", "delay", "hello", message -> {
        message.getMessageProperties().setDelay(5000); // 延迟 5 秒
        return message;
    });
}
```

### 测试效果

```
消息发送：约 21:38:49
消息接收：21:38:54
延迟：约 5 秒 ✅
```

## 五、方案对比

| 对比项 | TTL + 死信队列 | DelayExchange 插件 |
|---|---|---|
| 实现复杂度 | 需要额外声明死信交换机/队列 | 直接在交换机上设置延迟 |
| 时间精度 | 不准确（队首处理） | 精确 |
| 消息流转 | 消息 → 普通队列 → 死信队列 → 消费 | 消息 → 延迟交换机 → 直接到队列 → 消费 |
| 长延迟支持 | 可以但不推荐 | 不推荐（CPU 开销大） |
| 适用场景 | 简单延迟、对精度要求不高 | 短延迟、对精度要求高 |

### 选型建议

| 场景 | 推荐方案 |
|---|---|
| 短延迟（几秒~几分钟） | DelayExchange 插件 ✅ |
| 长延迟（几小时~几天） | 定时任务轮询数据库 |
| 超时取消订单 | 短超时用插件，长超时用定时任务 |

## 六、注意事项

### DelayExchange 插件的限制

延迟插件内部维护本地数据库表，使用 Erlang Timers 实现计时：
- 延迟时间过长 → 堆积的延迟消息多 → CPU 开销大
- 延迟时间会有误差
- **不建议设置延迟时间过长的延迟消息**

### 兜底方案

虽然用了延迟消息，但也不能保证 100% 可靠。兜底方案：**定时任务定期查询订单状态**。

```
交易服务 → 每隔20秒查询未支付订单 → 超时未支付 → 取消订单
```

**一句话总结**：延迟消息 + 定时任务兜底 = 可靠的超时处理
