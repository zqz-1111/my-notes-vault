# 设备管理模块改造

## 一、写在前面

1. 为什么学这个：设备管理是核心业务模块，涉及设备类型、设备状态监控、策略分配、货道管理等多个子功能

## 二、正文

### 2.1 设备类型管理改造

#### 搜索精简

只保留型号名称搜索：

```html
<el-form-item label="型号名称" prop="name">
  <el-input v-model="queryParams.name" placeholder="请输入型号名称" clearable @keyup.enter="handleQuery" />
</el-form-item>
```

#### el-input-number 数字输入框

货道行/列和容量用数字输入框，带单位后缀：

```html
<el-form-item label="货道数" prop="vmRow">
  <el-input-number v-model="form.vmRow" placeholder="请输入" :min="1" :max="10"/>行&nbsp;&nbsp;
  <el-input-number v-model="form.vmCol" placeholder="请输入" :min="1" :max="10"/>列
</el-form-item>

<el-form-item label="货道容量" prop="channelMaxCapacity">
  <el-input-number v-model="form.channelMaxCapacity" placeholder="请输入" :min="1" :max="10"/>个
</el-form-item>
```

**关键点**：
- `:min="1"` 设置最小值，防止输入 0 或负数
- `:max="10"` 设置最大值，限制合理范围
- 单位文字（行、列、个）直接写在组件后面

### 2.2 设备管理改造

#### 表格中 v-for 匹配显示名称

后端只返回 ID（如 vmTypeId、partnerId），需要在前端通过 v-for 循环匹配显示名称：

```html
<el-table-column label="设备型号" align="center" prop="vmTypeId">
  <template #default="scope">
    <div v-for="item in vmTypeList" :key="item.id">
      <span v-if="item.id==scope.row.vmTypeId">{{ item.name }}</span>
    </div>
  </template>
</el-table-column>
```

**适用场景**：后端没有做关联查询，只返回了 ID 字段，前端需要自行匹配显示名称。

#### 对话框条件渲染（新增 vs 编辑）

根据 `form.innerCode` 是否为 null 区分新增和编辑模式：

```html
<!-- 新增时：系统自动生成编号 -->
<el-form-item label="设备编号">
  <span>{{ form.innerCode == null ? '系统自动生成' : form.innerCode }}</span>
</el-form-item>

<!-- 编辑时才显示的字段 -->
<el-form-item label="供货时间" v-if="form.innerCode != null">...</el-form-item>
<el-form-item label="设备类型" v-if="form.innerCode != null">...</el-form-item>
<el-form-item label="设备容量" v-if="form.innerCode != null">...</el-form-item>
<el-form-item label="合作商" v-if="form.innerCode != null">...</el-form-item>
<el-form-item label="所属区域" v-if="form.innerCode != null">...</el-form-item>
<el-form-item label="设备地址" v-if="form.innerCode != null">...</el-form-item>

<!-- 新增时才显示的字段 -->
<el-form-item label="选择型号" prop="vmTypeId" v-if="form.innerCode == null">
  <el-select v-model="form.vmTypeId" placeholder="请选择设备型号" style="width: 100%">
    <el-option v-for="item in vmTypeList" :key="item.id" :label="item.name" :value="item.id" />
  </el-select>
</el-form-item>
```

**关键点**：
- 新增模式（innerCode == null）：显示型号选择下拉
- 编辑模式（innerCode != null）：显示只读的设备信息
- 点位选择在两种模式下都显示（可修改）

### 2.3 设备策略分配

点击"策略"按钮弹出策略选择对话框：

```javascript
const policyOpen = ref(false);

function handleUpdatePolicy(row) {
  form.value.id = row.id;
  form.value.policyId = row.policyId;
  listPolicy(loadAllParams).then(response => {
    policyList.value = response.rows;
    policyOpen.value = true;
  });
}
```

对话框中用 el-select 下拉选择策略：

```html
<el-dialog title="策略管理" v-model="policyOpen" width="500px" append-to-body>
  <el-form ref="vmRef" :model="form" label-width="80px">
    <el-form-item label="策略" prop="policyId">
      <el-select v-model="form.policyId" placeholder="请选择策略">
        <el-option v-for="item in policyList" :key="item.policyId" :label="item.policyName" :value="item.policyId" />
      </el-select>
    </el-form-item>
  </el-form>
</el-dialog>
```

**关键点**：
- 取消按钮需要同时关闭两个对话框（`open` 和 `policyOpen`）
- `handleUpdatePolicy` 直接用 `form.value.id = row.id` 赋值，不调用 reset()，保留设备信息

### 2.4 设备状态页面

设备状态是独立于设备管理的页面（vmStatus/index.vue），专注于查看设备运行状态，不需要增删改功能。

#### 运营状态 vs 设备状态

两个不同的概念，用不同方式展示：

```html
<!-- 运营状态：使用字典标签，如 未投放/运营/撤机 -->
<el-table-column label="运营状态" align="center" prop="vmStatus">
  <template #default="scope">
    <dict-tag :options="vm_status" :value="scope.row.vmStatus"/>
  </template>
</el-table-column>

<!-- 设备状态：解析 JSON 字符串，判断设备是否正常运行 -->
<el-table-column label="设备状态" align="center" prop="vmStatus">
  <template #default="scope">
    {{ scope.row.runningStatus != null ? JSON.parse(scope.row.runningStatus).status == true ? '正常' : '异常' : '异常' }}
  </template>
</el-table-column>
```

**关键点**：
- `runningStatus` 是 JSON 字符串，存储设备硬件状态（如 `{"status": true}`）
- 用 `JSON.parse()` 解析后取 `status` 字段
- 嵌套三元运算符：先判空，再判 status 值

### 2.5 货道管理（父子组件通信）

#### 父组件调用子组件

父组件（vm/index.vue）通过 props 传递数据，通过事件接收子组件通知：

```html
<el-button link type="primary" @click="handleGoods(scope.row)">货道</el-button>
<ChannelDialog :goodVisible="goodVisible" :goodData="goodData" @handleCloseGood="handleCloseGood" />
```

```javascript
import ChannelDialog from './components/ChannelDialog.vue';

const goodVisible = ref(false);
const goodData = ref({});

function handleGoods(row) {
  goodVisible.value = true;
  goodData.value = row; // 传递整行数据（含 innerCode、vmTypeId）
}
function handleCloseGood() {
  goodVisible.value = false;
}
```

#### 子组件接收与通知

子组件（ChannelDialog.vue）用 `defineProps` 接收、`defineEmits` 发送事件：

```javascript
const props = defineProps({
  goodVisible: { type: Boolean, default: false },
  goodData: { type: Object, default: () => {} },
});
const emit = defineEmits(['handleCloseGood']);

// 监听 props 变化，同步到本地变量
watch(() => props.goodVisible, (val) => {
  visible.value = val;
});

// 关闭时通知父组件
const handleClose = () => {
  visible.value = false;
  emit('handleCloseGood');
};
```

#### 子组件弹窗打开时加载数据

```javascript
const handleGoodOpen = async () => {
  const { data } = await getGoodsType(props.goodData.vmTypeId);
  vmType.value = data;
  const { data: channelData } = await getGoodsList(props.goodData.innerCode);
  channels.value = channelData;
};
```

## 三、方案对比

### 前端匹配名称方案

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| v-for 循环匹配 | 简单直接，无需后端改动 | 数据量大时性能差 | 下拉列表数据量小（<100） |
| 后端关联查询 | 前端简单，性能好 | 需要后端改动 | 数据量大或嵌套层级深 |

## 四、踩坑避坑指南

**坑 1：样式文件注释语法**

```vue
// ❌ 错误：JS 注释语法放在 </script> 后面无效
// <style lang="scss" src="../index.scss"></style>

<!-- ✅ 正确：HTML 注释 -->
<!-- <style lang="scss" src="../index.scss"></style> -->
```

**坑 2：scoped 样式对子组件的影响**
- 父组件引入样式不带 `scoped`，子组件可以继承
- 但子组件应该自己引入所需样式，保持独立性
- `:deep()` 选择器在 scoped 下行为不同，需要注意

**坑 3：el-input-number 的精度问题**
- `:precision="2"` 控制显示精度，但底层值可能有浮点误差
- 价格转换时用 `Math.round()` 避免 `0.1 * 100 = 10.000000000000002`

**坑 4：API 路径参数 vs 查询参数**

```javascript
// ❌ 错误：把 innerCode 拼到 URL 路径里
url: '/manage/channel/list/' + innerCode
// 实际请求：GET /manage/channel/list/A1000001 → 404

// ✅ 正确：用 params 传查询参数
url: '/manage/channel/list',
params: { innerCode }
// 实际请求：GET /manage/channel/list?innerCode=A1000001
```

后端 `@GetMapping("/list")` 没有路径变量，前端必须用 `params` 传参。

**坑 5：ref 初始化类型不匹配**

```javascript
// ❌ 错误：API 返回数组，但初始化为对象
const channels = ref({}); // channels[0] → undefined

// ✅ 正确：初始化为数组
const channels = ref([]); // channels[0] → 正常取值
```

模板中 `channels[index]` 访问对象的数字索引返回 undefined，访问数组的数字索引才正常。

**坑 6：async 出错导致 loading 永远转圈**

```javascript
// ❌ 错误：没有 try-catch，报错时 listLoading.value = false 不会执行
const channelList = async () => {
  listLoading.value = true;
  const { data } = await getGoodsList(innerCode); // 抛异常
  channels.value = data;
  listLoading.value = false; // 永远到不了这里
};

// ✅ 正确：用 try-catch-finally 确保 loading 关闭
const channelList = async () => {
  listLoading.value = true;
  try {
    const { data } = await getGoodsList(innerCode);
    channels.value = data || [];
  } catch (e) {
    channels.value = [];
  } finally {
    listLoading.value = false; // 无论成功失败都会执行
  }
};
```

**坑 7：子组件 prop 为 undefined 时模板报错**

父组件传入 `channels[index]`，如果 index 越界或 channels 未加载完，传入 undefined，子组件访问 `channel.channelCode` 就会报错。解决方法：
- 父组件确保初始化为数组
- 子组件模板加可选链：`channel?.channelCode`
- 或用 `v-if="channel"` 包裹整个组件
