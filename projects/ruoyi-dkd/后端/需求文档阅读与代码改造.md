# 需求文档阅读与后端代码改造

## 一、写在前面

1. 为什么学这个：了解项目的业务需求，明确哪些代码需要手写、哪些可以自动生成

## 二、项目业务概述

帝可得（DKD）是一个**智能售货机运营管理系统**，核心业务模块：

| 模块 | 核心表 | 关键功能 |
|---|---|---|
| 点位管理 | tb_region, tb_partner, tb_node | 区域、合作商、点位的 CRUD |
| 人员管理 | tb_emp, tb_role | 员工关联角色和区域 |
| 设备管理 | tb_vm_type, tb_vending_machine, tb_channel | 设备类型、设备、货道管理 |
| 策略管理 | tb_policy | 策略分配给设备，用于折扣营销 |
| 商品管理 | tb_sku_class, tb_sku | 商品类型、商品 CRUD，货道关联商品 |
| 工单管理 | tb_task, tb_task_details | 运维工单（投放/撤机/维修）+ 运营工单（补货） |

## 四、区域管理改造

### 4.1 需求

在区域列表查询中，需要显示每个区域的**点位数**。

### 4.2 实现思路

采用**关联查询**方案（区域和点位表记录都不多，实时查询数据准确）：

```sql
-- 先聚合统计每个区域的点位数，然后与区域表关联查询
select r.id, r.region_name, r.remark, ifnull(n.node_count, 0) as node_count 
from tb_region r
left join (select region_id, count(*) as node_count from tb_node group by region_id) n 
on r.id = n.region_id
```

### 4.3 代码实现

**RegionVo.java** — 新增点位数字段：

```java
@Data
public class RegionVo extends Region {
    // 点位数量
    private int nodeCount;
}
```

**RegionMapper.xml** — 关联查询 SQL：

```xml
<select id="selectRegionVoList" resultType="com.dkd.manage.domain.vo.RegionVo">
    select r.id, r.region_name, r.remark, ifnull(n.node_count, 0) as node_count 
    from tb_region r
    left join (select region_id, count(*) as node_count from tb_node group by region_id) n 
    on r.id = n.region_id
    <where>
        <if test="regionName != null and regionName != ''">
            and r.region_name like concat('%', #{regionName}, '%')
        </if>
    </where>
</select>
```

**RegionController.java** — 列表接口改用 Vo 方法：

```java
@GetMapping("/list")
public TableDataInfo list(Region region) {
    startPage();
    List<RegionVo> voList = regionService.selectRegionVoList(region);
    return getDataTable(voList);
}
```

## 五、合作商管理改造

### 5.1 合作商列表（显示点位数）

与区域列表实现方式相同，关联查询显示每个合作商的点位数：

```xml
<select id="selectPartnerVoList" resultType="com.dkd.manage.domain.vo.PartnerVo">
    SELECT p.*, COUNT(n.id) AS node_count FROM tb_partner p
    LEFT JOIN tb_node n ON p.id = n.partner_id
    <where>
        <if test="partnerName != null and partnerName != ''">
            and partner_name like concat('%', #{partnerName}, '%')
        </if>
    </where>
    GROUP BY p.id
</select>
```

### 5.2 新增合作商（密码加密）

新增合作商时，使用 `SecurityUtils.encryptPassword` 对密码加密：

```java
@Override
public int insertPartner(Partner partner) {
    // 使用SecurityUtil工具类，对密码加密
    partner.setPassword(SecurityUtils.encryptPassword(partner.getPassword()));
    partner.setCreateTime(DateUtils.getNowDate());
    return partnerMapper.insertPartner(partner);
}
```

## 六、踩坑避坑指南

### 坑 1：类型不兼容错误

**报错信息**：`不兼容的类型。实际为 java.util.List<PartnerVo>，需要 java.util.List<Partner>`

**原因分析**：Controller 的 export 方法用 `List<Partner>` 接收，但 service 方法返回的是 `List<PartnerVo>`

**解决方案**：
- 列表查询 → 用 `selectPartnerVoList`（返回 `List<PartnerVo>`，带点位数）
- 导出功能 → 用 `selectPartnerList`（返回 `List<Partner>`，原始数据）

### 坑 2：Mapper 方法缺失

**报错信息**：`无法解析 'RegionMapper' 中的方法 'selectRegionList'`

**原因分析**：Mapper 接口中缺少方法声明，Service 调用会报错

**解决方案**：在 RegionMapper.java 中补上方法声明

```java
public List<Region> selectPartnerList(Partner partner);
```

## 七、全文总结

1. 若依代码生成器能生成 70% 的基础代码，我们专注写 30% 的业务逻辑
2. 关联查询是常见的改造点，用 LEFT JOIN + 子查询实现统计
3. 导出功能不需要关联数据时，用原始的 selectList 方法即可
4. Mapper 接口、Service 接口、Service 实现、Controller 四层要保持一致

## 九、点位管理改造（NodeController）

### 9.1 需求

点位列表接口需要返回关联数据（区域名称、合作商名称、设备数量），使用 `NodeVo` 替代原始 `Node`。

### 9.2 实现思路

与区域/合作商改造类似，把 `selectNodeList` 改为 `selectNodeVoList`，返回带关联信息的 `NodeVo`。

### 9.3 踩坑：重复方法导致编译错误

**问题**：文件底部多了一个同名的 `list` 方法，两个方法都是 `@GetMapping("/list")` + `list(Node node)` 签名。

**后果**：Spring 不允许同一路径映射两个方法，编译报错。

**正确做法**：修改原有的 `list` 方法体，删掉底部重复的。一个 Controller 里同一个路径只能有一个映射方法。

## 十、MySQL 外键：NO ACTION vs RESTRICT

### 10.1 知识点

在 MySQL InnoDB 引擎中，**`NO ACTION` 和 `RESTRICT` 是完全等价的**，行为一模一样：如果被引用的记录存在，不允许删除或更新父表记录。

### 10.2 Navicat 显示问题

设置了 `NO ACTION` 后，Navicat 可能显示为 `RESTRICT`，这不是配置问题，是 Navicat 的展示方式。

### 10.3 验证方法

用命令行查询外键规则才是最准确的：

```sql
SELECT kcu.CONSTRAINT_NAME, kcu.COLUMN_NAME, kcu.REFERENCED_TABLE_NAME, 
       rc.UPDATE_RULE, rc.DELETE_RULE 
FROM information_schema.KEY_COLUMN_USAGE kcu 
JOIN information_schema.REFERENTIAL_CONSTRAINTS rc 
  ON kcu.CONSTRAINT_SCHEMA = rc.CONSTRAINT_SCHEMA 
  AND kcu.CONSTRAINT_NAME = rc.CONSTRAINT_NAME 
WHERE kcu.TABLE_SCHEMA='dkd' AND kcu.TABLE_NAME='tb_node';
```
