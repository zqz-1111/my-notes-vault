# 【Service层】新增/修改时补充冗余字段（区域名称 + 角色信息）

## 一、写在前面

1. **解决什么问题**：`tb_emp` 表里存了 `regionId` 和 `roleId` 两个外键，但列表展示时需要显示「区域名称」和「角色名称/编码」。如果每次都 JOIN 查询，列表接口会变慢。
2. **学完你能掌握**：在 Service 层的新增/修改方法中，通过外键查出冗余字段直接写入数据库，查询时无需关联。

## 二、核心思路

前端只传 `regionId`（Long）和 `roleId`（Long），后端在入库前自动补全：

- `regionId` → 查出 `regionName`（String）
- `roleId` → 查出 `roleName`（String）+ `roleCode`（String）

这样数据库里同时存了 ID 和名称，查询列表时直接 `SELECT *` 就能拿到所有展示字段，**用空间换时间**。

## 三、代码实现

### 3.1 注入依赖的 Mapper

```java
@Autowired
private RegionMapper regionMapper;

@Autowired
private RoleMapper roleMapper;
```

> 需要额外 import：`RegionMapper`、`RoleMapper`、`Role`

### 3.2 新增方法 insertEmp

```java
@Override
public int insertEmp(Emp emp)
{
    // 补充区域名称（通过 regionId 查 Region 表）
    emp.setRegionName(regionMapper.selectRegionById(emp.getRegionId()).getRegionName());
    // 补充角色信息（通过 roleId 查 Role 表）
    Role role = roleMapper.selectRoleByRoleId(emp.getRoleId());
    emp.setRoleName(role.getRoleName());
    emp.setRoleCode(role.getRoleCode());
    emp.setCreateTime(DateUtils.getNowDate());
    return empMapper.insertEmp(emp);
}
```

### 3.3 修改方法 updateEmp

```java
@Override
public int updateEmp(Emp emp)
{
    // 同样的补全逻辑
    emp.setRegionName(regionMapper.selectRegionById(emp.getRegionId()).getRegionName());
    Role role = roleMapper.selectRoleByRoleId(emp.getRoleId());
    emp.setRoleName(role.getRoleName());
    emp.setRoleCode(role.getRoleCode());
    emp.setUpdateTime(DateUtils.getNowDate());
    return empMapper.updateEmp(emp);
}
```

## 四、涉及的文件

| 文件 | 改动 |
|---|---|
| `EmpServiceImpl.java` | 新增注入 `RegionMapper`/`RoleMapper`，insert 和 update 方法补全字段 |
| `Emp.java` | 无改动，已有 `regionName`、`roleName`、`roleCode` 字段 |
| `RegionMapper.java` | 无改动，已有 `selectRegionById` 方法 |
| `RoleMapper.java` | 无改动，已有 `selectRoleByRoleId` 方法 |

## 五、踩坑避坑

### 逻辑坑：空指针风险

- **问题**：如果传入的 `regionId` 或 `roleId` 在数据库里不存在，`.getRegionName()` 会空指针
- **后果**：接口 500 报错
- **当前做法**：前端下拉选择保证 ID 合法，暂未加校验
- **更稳的做法**：加 null 判断或用 `Optional`，或在 Controller 层用 `@Validated` 校验 ID 必填

### 设计坑：冗余数据一致性

- **问题**：区域名称改了，`tb_emp` 里的 `regionName` 不会自动同步
- **后果**：列表显示旧名称
- **解决思路**：区域改名时同步更新关联的 emp 记录，或者干脆不冗余、查询时 JOIN（视业务量级决定）

## 六、方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 冗余存储（当前方案） | 查询快，无需 JOIN | 数据可能不一致，写入多一步 | 数据变更少、查询频繁 |
| 查询时 JOIN | 数据永远一致 | 查询慢，SQL 复杂 | 数据频繁变更、数据量小 |
| 查询时代码关联 | SQL 简单 | N+1 查询问题 | 数据量极小 |

**选择**：项目数据量不大，区域/角色名称变更频率低，冗余存储是合理的。

## 七、总结

- Service 层可以在入库前通过外键查表，把名称等冗余字段一并写入
- 核心就是「用空间换时间」——数据库多存几个字段，查询时少一次 JOIN
- 注意空指针和数据一致性两个坑
