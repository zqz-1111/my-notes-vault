# 设备修改空值判断 + 策略模块自动生成

## 一、写在前面

1. 为什么学这个：修改接口和新增接口的逻辑差异——新增时字段必填，修改时字段可能为 null
2. 学完这篇你能掌握：修改接口做空值判断的必要性 + 若依代码生成器的使用

## 二、设备修改接口优化

### 2.1 问题背景

`updateVendingMachine` 方法直接查询点位表补充信息，但如果前端只修改了设备名称（没传 `nodeId`），就会空指针。

### 2.2 解决方案：加 null 判断

```java
@Override
public int updateVendingMachine(VendingMachine vendingMachine)
{
    if (vendingMachine.getNodeId()!=null) {
        // 查询点位表，补充：区域、点位、合作商等信息
        Node node = nodeService.selectNodeById(vendingMachine.getNodeId());
        BeanUtil.copyProperties(node,vendingMachine,"id");// 商圈类型、区域、合作商
        vendingMachine.setAddr(node.getAddress());// 设备地址
    }
    vendingMachine.setUpdateTime(DateUtils.getNowDate());// 更新时间
    return vendingMachineMapper.updateVendingMachine(vendingMachine);
}
```

### 2.3 新增 vs 修改的区别

| 场景 | nodeId 是否必填 | 是否需要 null 判断 |
|---|---|---|
| 新增设备 | 必填（业务要求） | 不需要 |
| 修改设备 | 可选（可能只改其他字段） | 需要 |

**核心逻辑：** 新增时必须关联点位，所以 `nodeId` 一定有值；修改时可能只改状态、名称等字段，`nodeId` 可能为 null。

## 三、若依代码生成器

### 3.1 生成了什么

这次用若依代码生成器生成了**策略管理（Policy）**模块，包括：

**后端文件：**
- `PolicyController.java` — 控制器（CRUD 接口 + 权限注解 + 日志注解）
- `Policy.java` — 实体类（policyId, policyName, discount）
- `IPolicyService.java` — 服务接口
- `PolicyServiceImpl.java` — 服务实现
- `PolicyMapper.xml` — MyBatis 映射

**前端 API：**
- `policy.js` — 策略管理 API
- `vm.js` — 设备管理 API
- `vmType.js` — 设备类型 API

### 3.2 生成的代码结构

若依生成器自动生成的标准 CRUD 代码：
- 列表查询（分页 + 条件筛选）
- 详情查询
- 新增（自动补充 createTime）
- 修改（自动补充 updateTime）
- 删除（单条 + 批量）

### 3.3 策略表字段

| 字段 | 类型 | 说明 |
|---|---|---|
| policy_id | Long | 策略 ID（主键） |
| policy_name | String | 策略名称 |
| discount | Long | 折扣方案（如 80 代表 8 折） |
| create_time | Date | 创建时间 |
| update_time | Date | 更新时间 |

## 四、踩坑避坑指南

### 逻辑坑

**坑 1：修改接口不做空值判断**
- 问题描述：直接用 `nodeId` 查数据库，没判断是否为 null
- 后果：前端只改部分字段时，空指针异常
- 正确做法：加 `if (vendingMachine.getNodeId()!=null)` 判断

## 五、全文总结

- 修改接口和新增接口的核心区别：字段是否必填
- 若依代码生成器能自动生成 70% 的 CRUD 代码，只需要补充业务逻辑
- 策略管理模块是新的业务模块，后续需要和设备管理关联
