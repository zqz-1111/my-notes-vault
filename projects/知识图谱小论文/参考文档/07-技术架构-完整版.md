## 二、数据层详细设计

### 2.1 数据源清单

| 数据源 | 内容 | 用途 | 获取方式 |
|---|---|---|---|
| TCMNER2026 | 221,318实体 / 6,819病历 / 11类实体 | NER训练 | 邮件申请（已发） |

| TCM文献 | 中医临床文本、古籍、处方 | 补充语料 | 公开资源 |

### 2.2 实体类型定义（6类）

| 实体类型 | 英文 | 说明 | 示例 | 来源 |
|---|---|---|---|---|
| 中药 | Herb | 单味中药 | 黄芪、当归 | TCMNER2026(CMM) |
| 方剂 | Formula | 中药处方 | 四物汤、六味地黄丸 | TCMNER2026(TCMP) |
| 成分 | Component | 化学成分 | 槲皮素、黄芩苷 | TCMSP |
| 疾病 | Disease | 疾病名称 | 冠心病、糖尿病 | TCMNER2026(TCMDM/WMDT) |
| 症状 | Symptom | 临床症状 | 气短、乏力 | TCMNER2026(TCMS/WMS) |
| 靶点 | Target | 蛋白靶点 | PTGS2、ESR1 | TCMSP |

### 2.3 关系类型定义（5类）

| 关系类型 | 头实体 | 尾实体 | 示例 | 来源 |
|---|---|---|---|---|
| contains | Formula | Herb | 四物汤 → contains → 当归 | TCM_KG |
| has_component | Herb | Component | 黄芪 → has_component → 槲皮素 | TCMSP |
| treats | Herb/Formula | Disease | 黄芪 → treats → 气虚 | TCM_KG |
| targets | Component | Target | 槲皮素 → targets → PTGS2 | TCMSP |
| indicates | Disease | Symptom | 气虚 → indicates → 乏力 | TCM_KG |

### 2.4 药理学特征（来自TCMSP）

| 特征 | 含义 | 类型 | 用途 |
|---|---|---|---|
| OB | 口服生物利用度（%） | 连续值 | 衡量药物吸收能力 |
| DL | 类药性（0-1） | 连续值 | 衡量成药可能性 |
| MW | 分子量 | 连续值 | 分子大小 |
| AlogP | 脂溶性 | 连续值 | 亲脂性 |
| Hdon | 氢键供体数 | 整数 | 分子间作用力 |
| Hacc | 氢键受体数 | 整数 | 分子间作用力 |

**特征编码方案**：
- 6个数值特征 → 归一化 → 全连接层 → 64维药理特征向量 v_pharm
- 每个中药/成分实体对应一个v_pharm

## 三、NER模块（创新一：词汇知识增强）

### 3.1 模型结构

```
输入：中医临床文本 "黄芪补气固表治疗气虚乏力"
        ↓
BERT编码器（bert-base-chinese）
        ↓ [batch_size, seq_len, 768]
        ↓                    ↓
SoftLexicon词典匹配          原始字向量
（中医词典 → BMES四元组）      (768维)
        ↓                    ↓
词向量拼接+LayerNorm          ↓
        ↓                    ↓
    Concat ──────────────────┘
        ↓ [batch_size, seq_len, 768+词向量维度]
    线性层 → K维发射分数
        ↓
    解码层（CRF 或 Span分类，见3.2）
        ↓
    实体提取（类型 + 边界）
```

### 3.2 解码层选择（CRF vs Span）

默认采用 **CRF** 作为解码层（与已有BERT-CRF实验一致）。若实验中发现 Span-based 方案效果更优，则替换解码层，不影响整体框架（SoftLexicon增强部分不变）。

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| CRF | 经典稳定，转移矩阵建模标签依赖 | 无法处理嵌套实体 | 序列标注首选 |
| Span分类 | 可处理嵌套实体，枚举所有span | 计算量O(n²) | 嵌套实体场景 |
| GlobalPointer | 基于全局指针，效率高 | 实现复杂度稍高 | 多实体类型 |

**策略**：先用CRF跑通全流程，再用Span/GlobalPointer做NER对比实验，选择F1更高的方案。

### 3.3 SoftLexicon细节

- **词典来源**：TCM_KG的实体词表(~17,000词) + 中医专业词典
- **匹配方式**：对每个字位置，查找所有包含该字的词，构建B/M/E/S四元组
- **权重**：TF-IDF权重衰减，罕见专业词权重放大
- **不用jieba**：硬切分会破坏嵌套结构，错误传播

### 3.4 训练策略

- **损失函数**：CRF负对数似然（CRF方案）/ 交叉熵（Span方案）
- **优化器**：AdamW, lr=2e-5（BERT层）/ 1e-3（解码层）
- **Batch size**：32
- **Epochs**：10-20
- **对抗训练**（可选）：FGM在Embedding层加扰动，增强鲁棒性

### 3.5 评价指标

- Entity-level Precision / Recall / F1（精确匹配：span边界+类型都对才算对）
- 按实体类型分别报告F1

## 四、LLM辅助RE数据构建

### 4.1 为什么需要这一步

- 目前没有公开的中医RE标注数据集
- 人工标注几千条关系三元组成本太高
- 方案：用LLM自动标注 + Self-consistency质量保证 + 人工抽检校验

### 4.2 数据构建流程

```
TCMNER2026文本
    ↓
NER模型识别实体
    ↓
实体两两配对（同句内的实体对）
    ↓
构建结构化Prompt
    ↓
LLM标注：Temperature=0 + 3次采样 + Majority Voting
    ↓
人工抽检校验（10-20%样本）
    ↓
RE训练数据集
```

### 4.3 Prompt设计（参考论文8+论文17）

```
系统角色：你是一个中医药领域知识图谱构建专家。

任务：判断以下两个实体之间的关系。

实体1（头实体）：{head_entity}，类型：{head_type}
实体2（尾实体）：{tail_entity}，类型：{tail_type}
上下文：{sentence}

关系类型定义：
1. contains：方剂包含中药（Formula → Herb）
2. has_component：中药含有成分（Herb → Component）
3. treats：中药/方剂治疗疾病（Herb/Formula → Disease）
4. targets：成分作用于靶点（Component → Target）
5. indicates：疾病表现为症状（Disease → Symptom）
6. no_relation：无明确关系

请输出JSON格式：{"relation": "关系类型", "confidence": 0-1, "reasoning": "推理过程"}
```

### 4.4 质量控制（Self-consistency策略）

- **多次采样**：每个样本用Temperature=0采样3次
- **Majority Voting**：3次结果取多数票，票数≥2的保留
- **置信度平均**：3次confidence取平均，保留均值 ≥ 0.8的标注
- **人工抽检**：随机抽取10-20%样本人工审核
- **目标规模**：3,000-5,000条RE训练样本

**论文表述**："采用Self-consistency策略（Wang et al., 2023）增强LLM标注质量，通过多次采样与多数投票降低单次推理的随机噪声。"

## 五、RE模块（创新二+三：药理知识+KG先验增强）

### 5.1 模型结构

```
输入：句子 + 头实体 + 尾实体
        ↓
BERT编码器（+Entity Marker标记实体位置）
        ↓
    [CLS]向量 + 头实体向量 + 尾实体向量
        ↓                    ↓                    ↓
    h_text (768维)      h_head (768维)        h_tail (768维)
        ↓                    ↓                    ↓
        └────────────────────┼────────────────────┘
                             ↓
                    实体对表示 (拼接/差积)
                             ↓
                    ┌────────┴────────┐
                    ↓                 ↓
            药理特征融合         KG先验融合
           (创新二)            (创新三)
                    ↓                 ↓
           ┌───────┴───────┐         ↓
           ↓               ↓         ↓
     v_pharm_head    v_pharm_tail   v_kg
     (64维)          (64维)        (128维)
           ↓               ↓         ↓
           └───────┬───────┘         ↓
                   ↓                 ↓
            药理注意力融合      KG注意力融合
                   ↓                 ↓
                   └────────┬────────┘
                            ↓
                   特征融合层 (Concat + Dense)
                            ↓
                   关系分类器 (Softmax)
                            ↓
                   关系类型输出
```

### 5.2 Entity Marker（实体标记）

在输入文本中用特殊标记标注实体位置，帮助BERT更好地捕捉实体边界：

```
原始：黄芪补气固表治疗气虚乏力
标记：[E1] 黄芪 [/E1] 补气固表治疗 [E2] 气虚 [/E2] 乏力
```

- `<e1>`, `</e1>`：头实体边界标记
- `<e2>`, `</e2>`：尾实体边界标记
- 标记加入BERT词表，随机初始化后随模型微调
- 实体向量取标记位置的隐藏状态（h_head = H[e1], h_tail = H[e2]）

### 5.3 药理特征融合（创新二）

**编码**：
- 从TCMSP查询实体对应的OB/DL/MW/AlogP/Hdon/Hacc
- 6个数值特征 → MinMax归一化 → 全连接层 → 64维向量 v_pharm

**融合方式**：
```
h_combined = [h_head; h_tail; h_head - h_tail; h_head * h_tail]
v_pharm = [v_pharm_head; v_pharm_tail]

# 注意力融合
attention_score = softmax(W_a * [h_combined; v_pharm])
h_attended = attention_score * v_pharm

# 最终表示
h_final = [h_combined; h_attended]
```

### 5.4 KG先验融合（创新三）

**图嵌入获取**：
- 在TCM_KG上预训练图嵌入（默认TransE，可替换RotatE/ComplEx，通过实验选择最优）
- 每个实体得到128维图嵌入向量 v_kg_entity

**子图提取**：
- 对于实体对(h, t)，提取TCM_KG中h和t的1跳邻居
- 构建局部子图

**融合方式**：
```
v_kg_head = lookup(h)  # 128维
v_kg_tail = lookup(t)  # 128维
v_kg = [v_kg_head; v_kg_tail]  # 256维

# 注意力融合
attention_score = softmax(W_k * [h_text; v_kg])
h_kg_attended = attention_score * v_kg

# 最终表示
h_final = [h_combined; h_pharm_attended; h_kg_attended]
```

### 5.5 分类器

```python
# 最终分类
h_final = concat(h_combined, h_pharm_attended, h_kg_attended)  # 拼接所有特征
h_hidden = ReLU(W_1 * h_final + b_1)  # 隐藏层
logits = W_2 * h_hidden + b_2  # 输出层
relation = softmax(logits)  # 关系概率
```

### 5.6 训练策略

- **损失函数**：交叉熵（类别加权，处理no_relation长尾）
- **优化器**：AdamW, lr=2e-5
- **Batch size**：16-32
- **Epochs**：10-20
- **早停**：验证集F1连续3个epoch不提升则停止

### 5.7 评价指标

- Macro-F1 / Micro-F1
- 按关系类型分别报告P/R/F1
- 对比实验：BERT baseline / BERT+药理 / BERT+KG / 完整模型

## 六、知识图谱构建

### 6.1 流程

```
NER识别的实体 + RE预测的关系
        ↓
三元组去重（同实体同关系只保留一次）
        ↓
实体对齐（同义词合并，如"黄茋"→"黄芪"）
        ↓
Neo4j写入（CREATE实体节点 + CREATE关系边）
        ↓
Pyvis可视化 + Cypher查询验证
```

### 6.2 Neo4j设计

**节点属性**：
```cypher
// 中药节点
CREATE (h:Herb {
  name: "黄芪",
  ob: 47.26,
  dl: 0.74,
  nature: "甘，微温",
  meridian: "脾，肺"
})

// 疾病节点
CREATE (d:Disease {
  name: "气虚"
})
```

**关系属性**：
```cypher
// 治疗关系
CREATE (h)-[:TREATS {
  confidence: 0.92,
  source: "TCM_KG"
}]->(d)
```

### 6.3 可视化

- Pyvis生成交互式HTML知识图谱
- 按实体类型着色（Herb=绿色，Disease=红色，Component=蓝色）
- 节点大小按度数（连接数）缩放

## 七、实验方案

### 7.1 基线模型

| 模型 | 说明 | 任务 |

| BERT | 纯BERT分类 | RE |

### 7.2 对比实验

| 方法 | NER | RE | 说明 |
|---|---|---|---|
| BiLSTM-CRF | ✓ | | 经典baseline |
| BERT-CRF | ✓ | | 预训练baseline |
| BERT-CRF+SoftLexicon | ✓ | | 创新一单独 |

### 7.3 消融实验

| 实验 | 去掉什么 | 验证什么 |
|---|---|---|
| w/o SoftLexicon | 词典增强 | 词汇知识对NER的贡献 |
| w/o 药理特征 | OB/DL融合 | 药理知识对RE的贡献 |
| w/o KG先验 | 图嵌入融合 | KG结构对RE的贡献 |
| w/o 注意力融合 | 改为直接拼接 | 注意力机制的效果 |
| w/o LLM数据增强 | 用少量人工标注 | LLM辅助标注的价值 |
| w/o Entity Marker | 去掉实体标记 | Entity Marker对RE的贡献 |

### 7.4 Error Analysis

对NER和RE的错误进行系统分析，按错误类型分类统计：

**NER错误类型**：
| 错误类型 | 说明 | 示例 |
|---|---|---|
| 边界错误 | 实体边界识别不准 | "补气固表"只识别了"补气" |
| 类型错误 | 实体类型判错 | 把"症状"标成"疾病" |
| 遗漏 | 未识别出实体 | 文本中有"黄芪"但没标出 |
| 古籍写法 | 古文表述未覆盖 | "耆"（黄芪古称）未识别 |

**RE错误类型**：
| 错误类型 | 说明 | 示例 |
|---|---|---|
| 跨句关系 | 头尾实体不在同一句 | 需要跨句推理 |
| 共指消解 | 同一实体不同表述 | "该药"指代前文"黄芪" |
| 多关系 | 两个实体间存在多种关系 | 黄芪-气虚：treats + indicates |
| 方向错误 | 头尾实体方向反了 | 把"contains"方向搞反 |

**呈现方式**：
- 错误分布饼图（各类型占比）
- 典型错误案例表格（每类2-3个例子）
- 错误原因分析 + 改进方向讨论
