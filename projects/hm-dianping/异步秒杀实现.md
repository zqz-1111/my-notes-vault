# 【秒杀】异步秒杀实现（Lua + 阻塞队列 + 异步下单）

## 一、写在前面

1. 为什么学这个 — 之前的秒杀方案是同步的，请求从判库存到写库全程串行，数据库成为瓶颈。异步秒杀把"判断能不能买"和"真正下单"拆开，同步只做 Redis 操作（微秒级），写库丢给异步线程（毫秒级），大幅提升并发能力
2. 学完这篇你能掌握 — Lua 脚本原子校验、阻塞队列、异步下单、锁的分层设计

## 二、需求总览

> 改进秒杀业务，提高并发性能

| 序号 | 需求 | 技术方案 |
|---|---|---|
| ① | 新增秒杀券时，库存写入 Redis | `StringRedisTemplate.set()` |
| ② | Lua 脚本原子判断库存 + 一人一单 | `DefaultRedisScript` + `seckill.lua` |
| ③ | 抢购成功，封装订单放入阻塞队列 | `ArrayBlockingQueue` |
| ④ | 开启异步线程，从队列消费，异步下单 | `@PostConstruct` + 线程池 |

## 三、整体架构

```
用户请求 seckillVoucher(voucherId)
    │
    ▼
┌──────────────────────────────────┐
│  Redis Lua 脚本（原子操作）        │
│  1. 查库存 ≤ 0? → 返回 1（库存不足）│
│  2. 用户已下单? → 返回 2（重复下单） │
│  3. 扣库存 + 记录用户 → 返回 0      │
└──────────────────────────────────┘
    │ 返回 0
    ▼
封装 VoucherOrder → 放入阻塞队列 → 立即返回 orderId
    │
    ▼ （异步线程，不阻塞用户）
从队列 take() → Redisson 分布式锁 → createVoucherOrder()
    │
    ▼
数据库层再判重 → CAS 扣库存 → 写入订单 → 释放锁
```

**核心思想：同步路径只操作 Redis（快），数据库写入丢给异步线程（慢）。**

## 四、逐步实现

### 4.1 需求①：新增秒杀券时，库存写入 Redis

**位置：** `VoucherServiceImpl.addSeckillVoucher()`

```java
// 保存秒杀库存到Redis中
stringRedisTemplate.opsForValue().set(SECKILL_STOCK_KEY + voucher.getId(), voucher.getStock().toString());
```

Redis 里的数据结构：
- `seckill:stock:{voucherId}` → `"200"`（库存数量，字符串类型）
- `seckill:order:{voucherId}` → `{userId1, userId2, ...}`（下单用户集合，Lua 里 `sadd` 自动创建）

> 为什么要存 Redis？因为后面 Lua 脚本要在 Redis 里读库存、扣库存，不走数据库，速度从毫秒降到微秒。

### 4.2 需求②：Lua 脚本原子校验

**位置：** `src/main/resources/seckill.lua`

```lua
-- 参数列表
local voucherId = ARGV[1]   -- 优惠券id
local userId = ARGV[2]      -- 用户id

-- 数据key
local stockKey = 'seckill:stock:' .. voucherId
local orderKey = 'seckill:order:' .. voucherId

-- 1.判断库存是否充足
if(tonumber(redis.call('get', stockKey)) <= 0) then
    return 1  -- 库存不足
end
-- 2.判断用户是否重复下单
if(redis.call('sismember', orderKey, userId) == 1) then
    return 2  -- 重复下单
end
-- 3.扣库存 + 记录用户
redis.call('incrby', stockKey, -1)
redis.call('sadd', orderKey, userId)
return 0  -- 成功
```

返回值含义：

| 返回值 | 含义 | Java 端处理 |
|---|---|---|
| 0 | 成功 | 继续封装订单，放入队列 |
| 1 | 库存不足 | 返回 `Result.fail("库存不足")` |
| 2 | 重复下单 | 返回 `Result.fail("不能重复下单")` |

> **为什么用 Lua？** 查库存 → 判重复 → 扣库存，三步必须原子。用 Java 分三步调 Redis，中间可能被其他请求插队。Lua 脚本在 Redis 里一次性执行完，天然原子。

### 4.3 需求③：抢购成功后放入阻塞队列

**位置：** `VoucherOrderServiceImpl.seckillVoucher()`

```java
// 定义阻塞队列（容量100万）
private BlockingQueue<VoucherOrder> orderTasks = new ArrayBlockingQueue<>(1024 * 1024);

public Result seckillVoucher(Long voucherId) {
    Long userId = UserHolder.getUser().getId();
    // 1.执行Lua脚本
    Long result = stringRedisTemplate.execute(SECKILL_SCRIPT, Collections.emptyList(),
            voucherId.toString(), userId.toString());
    int r = result.intValue();
    // 2.判断结果
    if (r != 0) {
        return Result.fail(r == 1 ? "库存不足" : "不能重复下单");
    }
    // 3.封装订单
    VoucherOrder voucherOrder = new VoucherOrder();
    long orderId = redisIdWorker.nextId("order");
    voucherOrder.setId(orderId);
    voucherOrder.setUserId(userId);
    voucherOrder.setVoucherId(voucherId);
    // 4.放入阻塞队列
    orderTasks.add(voucherOrder);
    // 5.返回订单id（不等写库）
    return Result.ok(orderId);
}
```

> 为什么立即返回？写数据库是慢操作（几十毫秒），但用户等不了那么久。这里只做了 Redis 操作（微秒级），直接告诉用户"你抢到了"，下单结果异步写库。

### 4.4 需求④：异步线程消费队列，真正下单

#### 4.4.1 线程启动（@PostConstruct）

```java
private static final ExecutorService SECKILL_ORDER_EXECUTOR = Executors.newSingleThreadExecutor();

@PostConstruct
private void init() {
    SECKILL_ORDER_EXECUTOR.submit(new VoucherOrderHandler());
    proxy = applicationContext.getBean(IVoucherOrderService.class);
}
```

`@PostConstruct` 在 Bean 初始化完成后执行，提交一个线程到线程池，永不停止地循环。

#### 4.4.2 从队列取任务

```java
private class VoucherOrderHandler implements Runnable {
    @Override
    public void run() {
        while (true) {
            try {
                VoucherOrder voucherOrder = orderTasks.take();  // 阻塞等待
                handleVoucherOrder(voucherOrder);
            } catch (Exception e) {
                log.error("处理订单异常", e);
            }
        }
    }
}
```

`take()` 会阻塞 —— 队列没数据时线程休眠，有数据时自动唤醒，不浪费 CPU。

#### 4.4.3 加分布式锁 + 写库

```java
private void handleVoucherOrder(VoucherOrder voucherOrder) {
    Long userId = voucherOrder.getUserId();
    RLock redisLock = redissonClient.getLock("lock:order:" + userId);
    boolean isLock = redisLock.tryLock();
    if (!isLock) {
        log.error("不允许重复下单！");
        return;
    }
    try {
        proxy.createVoucherOrder(voucherOrder);  // 用代理调用，保证事务生效
    } finally {
        redisLock.unlock();
    }
}

@Transactional
public void createVoucherOrder(VoucherOrder voucherOrder) {
    Long userId = voucherOrder.getUserId();
    // 再查一次数据库，防重
    int count = query().eq("user_id", userId).eq("voucher_id", voucherOrder.getVoucherId()).count();
    if (count > 0) {
        log.error("用户已经购买过了");
        return;
    }
    // CAS 扣库存
    boolean success = seckillVoucherService.update()
            .setSql("stock = stock - 1")
            .eq("voucher_id", voucherOrder.getVoucherId()).gt("stock", 0)
            .update();
    // 写入订单
    save(voucherOrder);
}
```

## 五、阻塞队列详解

### 5.1 什么是阻塞队列

**一个"生产者放、消费者取"的队列，取的时候如果没东西，线程会自动卡住等待，有东西了自动唤醒。**

生活类比：快递传送带
- 快递员（生产者）把包裹放上传送带 → `queue.put(包裹)`
- 你（消费者）从传送带上拿包裹 → `queue.take()`
- 传送带空了 → 你不会疯狂空转，而是原地等，包裹来了自动拿
- 传送带满了 → 快递员等你拿走一个再放

### 5.2 对比其他方式

| 方式 | 做法 | 问题 |
|---|---|---|
| 普通 List | `while(list.isEmpty()) {}` 死循环检查 | CPU 空转，浪费资源 |
| sleep 轮询 | `while(empty) { sleep(100) }` | 响应慢，间隔难调 |
| **BlockingQueue** | `queue.take()` 阻塞等待 | 线程休眠，有数据自动唤醒 |

### 5.3 常见实现类

| 类 | 特点 |
|---|---|
| `ArrayBlockingQueue` | 数组实现，**固定容量**，本项目用的这个 |
| `LinkedBlockingQueue` | 链表实现，可选容量，默认无限大 |
| `SynchronousQueue` | 不存数据，直接交接，容量为 0 |

## 六、锁的分工变化

### 6.1 之前（同步方案）

```
┌──────────────────────────────────────┐
│ Redisson锁                           │
│  判一人一单(DB) → 扣库存(DB) → 写订单 │
└──────────────────────────────────────┘
```

一把 Redisson 锁包住整个流程，从判重到写库全在锁里，串行执行。

### 6.2 现在（异步方案）

```
┌─────────────────┐     ┌──────────────────┐
│ Lua脚本(无锁)    │ →→→ │ Redisson锁        │
│ 判库存+判重+扣库存│ 队列 │ 写订单到DB         │
│ Redis原子操作     │     │ DB操作             │
└─────────────────┘     └──────────────────┘
    同步，微秒级              异步，毫秒级
```

### 6.3 为什么这么改

| 阶段 | 之前 | 现在 | 原因 |
|---|---|---|---|
| 判库存+判重+扣库存 | Redisson 锁 + 查 DB | **Lua 脚本** | Redis 原子操作，比加锁快 100 倍 |
| 写订单到数据库 | Redisson 锁 | **Redisson 锁** | 没变，DB 操作必须加锁 |

**Lua 脚本替掉了前半段锁的活，Redisson 锁只保护后半段写库。** 两层防护：Lua 在 Redis 层防一次，Redisson 锁在 DB 层再防一次，双重保险。

## 七、踩坑避坑指南

### 逻辑坑

**坑 1：Lua 脚本残留了 Redis Stream 代码**
- 问题：Lua 里有 `xadd stream.orders` 和 `ARGV[3]`（orderId），但 Java 端只传了 2 个参数
- 后果：orderId 在 Lua 执行后才生成，ARGV[3] 是 nil
- 正确做法：用 BlockingQueue 方案就删掉 Lua 里的 `xadd`，orderId 由 Java 端生成

**坑 2：proxy 获取方式错误**
- 问题：`proxy = AopContext.currentProxy()` 放在 `seckillVoucher` 方法体内，每次请求都重新赋值
- 后果：并发不安全，多线程同时写 proxy 字段
- 正确做法：在 `@PostConstruct` 里用 `applicationContext.getBean(IVoucherOrderService.class)` 一次性初始化

**坑 3：AopContext 在 @PostConstruct 里不能用**
- 问题：以为 `@PostConstruct` 里能调 `AopContext.currentProxy()`
- 后果：报错，因为此时没有 AOP 调用上下文
- 正确做法：用 `applicationContext.getBean()` 代替

**坑 4：为什么用 proxy.createVoucherOrder() 而不是 this.createVoucherOrder()**
- 问题：`@Transactional` 依赖 AOP 代理，直接调 `this.createVoucherOrder()` 事务不生效
- 后果：异步线程里扣库存和写订单不在同一个事务中
- 正确做法：提前在 `@PostConstruct` 里获取代理对象，用代理调用

### 报错坑

**坑 5：ClassPathResource 导错包**
- 报错：`'setLocation(Resource)' cannot apply to '(cn.hutool.core.io.resource.ClassPathResource)'`
- 原因：Hutool 和 Spring 都有 `ClassPathResource`，IDE 自动补全选错了
- 解决：改成 `org.springframework.core.io.ClassPathResource`

**坑 6：接口和实现类方法签名不匹配**
- 报错：`类"VoucherOrderServiceImpl"必须声明为抽象，或实现 'IVoucherOrderService' 中的抽象方法`
- 原因：重构实现类参数从 `(Long userId, Long voucherId)` 改成 `(VoucherOrder voucherOrder)`，接口忘同步
- 解决：接口也改成 `void createVoucherOrder(VoucherOrder voucherOrder)`

## 八、全文总结

- **同步路径只操作 Redis（快），数据库写入丢给异步线程（慢）**
- Lua 脚本保证判断+扣库存的原子性，替代了前半段的加锁方案
- BlockingQueue 实现生产者-消费者模型，`take()` 阻塞等待不浪费 CPU
- Redisson 锁只保护后半段 DB 写入，锁粒度缩小，并发能力提升
- 两层防护：Lua（Redis 层）+ Redisson 锁（DB 层），双重保险一人一单

## 九、进阶：用 Redis Stream 替换 BlockingQueue

> 本节是异步秒杀的第二阶段升级，将 JVM 内存的 BlockingQueue 替换为 Redis Stream 消息队列。

### 9.1 为什么要换

| 问题 | BlockingQueue | Redis Stream |
|---|---|---|
| 重启丢数据 | ❌ JVM 内存，重启就没了 | ✅ Redis 持久化 |
| 无法集群 | ❌ 每个 JVM 实例各自有队列 | ✅ 多实例竞争消费 |
| 消息确认 | ❌ 取走就没了 | ✅ XACK 确认机制 |
| 消息堆积 | ❌ 内存有限会阻塞 | ✅ Redis 存储，容量大 |

### 9.2 Redis 消息队列三种方案对比

| 特性 | List | Pub/Sub | Stream |
|---|---|---|---|
| 持久化 | ✅ | ❌ | ✅ |
| 消息确认（ACK） | ❌ 取走就没了 | ❌ | ✅ `XACK` |
| 消费者组 | ❌ | ❌ | ✅ |
| 阻塞读取 | ✅ `BRPOP` | ✅ 自动 | ✅ `BLOCK` |
| 历史消息回溯 | ❌ | ❌ | ✅ 按 ID 范围查 |
| 消息堆积 | ✅ | ❌ 不在线就丢 | ✅ |
| 适用场景 | 简单队列 | 广播通知 | 正式消息队列 |

**选择 Stream**：需要消费者组（竞争消费）+ ACK 确认（防丢）+ 持久化（重启不丢）。

> 详细对比见 [[Redis消息队列方案对比]]

### 9.3 改动总览

整体数据流变化：

```
之前（BlockingQueue）：
  Lua校验 → Java入队 → 内存队列取出 → 写DB

现在（Stream）：
  Lua校验+XADD → Redis Stream → XREADGROUP取出 → 写DB → XACK确认
```

改动点只有三个文件：

| 文件 | 改动 |
|---|---|
| `RedisConstants.java` | 新增 `STREAM_ORDER_KEY = "stream.orders"` |
| `seckill.lua` | 校验通过后增加 `XADD` 命令 |
| `VoucherOrderServiceImpl.java` | 删 BlockingQueue，改 Stream 消费者 |

### 9.4 Lua 脚本改动

```lua
-- 新增第三个参数：订单id（Java端提前生成传入）
local orderId = ARGV[3]

-- 校验通过后，直接往 Stream 推消息（原子操作）
redis.call('xadd', 'stream.orders', '*', 
    'userId', userId, 
    'voucherId', voucherId, 
    'orderId', orderId)
return 0
```

**关键变化**：orderId 从 Java 端提前生成，通过 ARGV[3] 传给 Lua。因为 Lua 里要把它写入消息，必须提前知道。

```java
// 之前：Lua校验 → Java生成orderId → Java入队
Long result = execute(lua, voucherId, userId);
long orderId = redisIdWorker.nextId("order");
orderTasks.add(voucherOrder);

// 现在：先生成orderId → Lua校验+入队（一步到位）
long orderId = redisIdWorker.nextId("order");
Long result = execute(lua, voucherId, userId, orderId);
```

**为什么在 Lua 里做 XADD？** 校验和入队是一个原子操作——要么全成功要么全失败。如果 Java 端做，Lua 校验过了但 Java 还没入队就崩了，订单就丢了。

### 9.5 消费者端改动

#### 删掉 BlockingQueue

```java
// 删掉这些
private BlockingQueue<VoucherOrder> orderTasks = new ArrayBlockingQueue<>(1024 * 1024);
// 删掉 orderTasks.add()
// 删掉 orderTasks.take()
```

#### @PostConstruct 初始化

```java
@PostConstruct
private void init() {
    proxy = applicationContext.getBean(IVoucherOrderService.class);
    // 1. 创建消费者组（stream不存在时先推init消息创建）
    try {
        stringRedisTemplate.opsForStream().createGroup(STREAM_ORDER_KEY, GROUP_NAME);
    } catch (Exception e) {
        // stream不存在 → 推一条初始化消息让stream自动创建 → 再建组
        stringRedisTemplate.opsForStream().add(STREAM_ORDER_KEY, Collections.singletonMap("init", "0"));
        stringRedisTemplate.opsForStream().createGroup(STREAM_ORDER_KEY, GROUP_NAME);
    }
    // 2. 启动消费者线程
    SECKILL_ORDER_EXECUTOR.submit(new VoucherOrderHandler());
}
```

#### 消费者线程（XREADGROUP）

```java
private class VoucherOrderHandler implements Runnable {
    @Override
    public void run() {
        while (true) {
            // 1. 从 Stream 读取消息（消费者组模式，阻塞2秒）
            List<MapRecord<String, Object, Object>> list = 
                stringRedisTemplate.opsForStream().read(
                    Consumer.from(GROUP_NAME, CONSUMER_NAME),
                    StreamReadOptions.empty().count(1).block(Duration.ofSeconds(2)),
                    StreamOffset.create(STREAM_ORDER_KEY, ReadOffset.lastConsumed())
                );
            if (list == null || list.isEmpty()) continue;

            // 2. 解析消息
            MapRecord<String, Object, Object> record = list.get(0);
            Map<Object, Object> value = record.getValue();
            
            // 跳过初始化消息
            if (value.get("orderId") == null) {
                opsForStream().acknowledge(STREAM_ORDER_KEY, GROUP_NAME, record.getId());
                continue;
            }
            
            VoucherOrder voucherOrder = new VoucherOrder();
            voucherOrder.setId(Long.valueOf(value.get("orderId").toString()));
            voucherOrder.setUserId(Long.valueOf(value.get("userId").toString()));
            voucherOrder.setVoucherId(Long.valueOf(value.get("voucherId").toString()));

            // 3. 处理订单（Redisson锁 + DB写入）
            handleVoucherOrder(voucherOrder);

            // 4. ACK确认（告诉Redis这条消息处理完了）
            opsForStream().acknowledge(STREAM_ORDER_KEY, GROUP_NAME, record.getId());
        }
    }
}
```

#### Pending 消息兜底

```java
        ...
    }
}
```

### 9.6 消息的生命周期

```
1. 用户请求 → Lua执行 → XADD 写入 stream.orders
   状态：pending（未被消费）

2. 消费者 XREADGROUP 读到消息
   状态：delivered（已投递，等待ACK）

3. handleVoucherOrder() 写库成功 → XACK 确认
   状态：已确认（从pending列表移除）

4. 如果步骤2-3之间崩了（没ACK）
   → 消息留在 pending 列表
   → 下次重启时 handlePendingList() 重新处理
```

### 9.7 Stream 核心命令速查

```bash
# 生产者
XADD stream.orders * userId 5 voucherId 13 orderId 123    # 发消息

# 消费者组
XGROUP CREATE stream.orders g1 0                           # 创建消费者组
XREADGROUP GROUP g1 c1 COUNT 1 BLOCK 2000 STREAMS stream.orders >   # 读新消息
XREADGROUP GROUP g1 c1 COUNT 1 STREAMS stream.orders 0              # 读pending消息
XACK stream.orders g1 消息ID                                # 确认消息

# 查询
XLEN stream.orders                                         # 消息总数
XPENDING stream.orders g1                                  # pending列表摘要
XRANGE stream.orders - +                                   # 查看所有消息
```

### 9.8 踩坑记录

**坑 1：stream 不存在时创建消费者组报错**
- 问题：`XGROUP CREATE` 要求 stream 已存在
- 解决：先推一条 init 消息让 stream 自动创建，再建组

**坑 2：init 消息没有 orderId 导致 NPE**
- 问题：初始化消息只有 `init:0`，消费者解析 `value.get("orderId")` 返回 null
- 解决：判断 `orderId == null` 则跳过并 ACK

**坑 3：直接往数据库插券，Redis 没有库存 key**
- 问题：Lua 脚本 `GET seckill:stock:{id}` 返回 nil，`nil <= 0` 比较报错
- 解决：必须通过 API 创建秒杀券（自动同步 Redis 库存 key）
