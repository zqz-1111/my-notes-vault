# Redis 分布式锁四大核心缺陷与优化方案

## 一、写在前面

基于 SETNX 实现的基础 Redis 分布式锁存在四大核心缺陷，是从入门实现升级为生产级分布式锁必须解决的问题。本文逐个分析问题本质和对应优化方案。

## 二、缺陷一：不可重入

### 问题描述

同一个线程无法多次获取同一把锁。

### 问题本质

基础 SETNX 实现仅判断 key 是否存在，没有记录锁的持有者身份和重入次数。在方法嵌套、递归调用等场景下，同一个线程二次加锁会直接失败，甚至引发死锁。

```java
// 场景：外层方法加锁，内层方法也需要同一把锁
public void outerMethod() {
    lock.lock();  // 第一次加锁成功
    innerMethod(); // 调用内层方法
    lock.unlock();
}

public void innerMethod() {
    lock.lock();  // ❌ 第二次加锁失败，因为锁已存在
    // ...
    lock.unlock();
}
```

### 优化方案：Redis Hash 结构

采用 Redis Hash 结构存储锁信息：
- **key**：锁名称
- **field**：线程唯一标识（客户端 ID + 线程 ID）
- **value**：重入次数

全程通过 Lua 脚本保证操作原子性：

**加锁 Lua 脚本：**
```lua
-- KEYS[1]=锁key, ARGV[1]=线程标识, ARGV[2]=过期时间
if redis.call('EXISTS', KEYS[1]) == 0 then
    -- 锁不存在，创建，重入次数=1
    redis.call('HSET', KEYS[1], ARGV[1], 1)
    redis.call('EXPIRE', KEYS[1], ARGV[2])
    return 1
elseif redis.call('HEXISTS', KEYS[1], ARGV[1]) == 1 then
    -- 锁存在且是自己的，重入次数+1
    redis.call('HINCRBY', KEYS[1], ARGV[1], 1)
    redis.call('EXPIRE', KEYS[1], ARGV[2])
    return 1
else
    -- 锁被别人持有，失败
    return 0
end
```

**释放锁 Lua 脚本：**
```lua
-- KEYS[1]=锁key, ARGV[1]=线程标识
if redis.call('HEXISTS', KEYS[1], ARGV[1]) == 0 then
    -- 锁不是自己的，不能释放
    return nil
end
-- 重入次数-1
local counter = redis.call('HINCRBY', KEYS[1], ARGV[1], -1)
if counter > 0 then
    -- 还有重入，刷新过期时间
    redis.call('EXPIRE', KEYS[1], ARGV[2])
    return 0
else
    -- 重入次数归零，真正删除锁
    redis.call('DEL', KEYS[1])
    return 1
end
```

> **Redisson 实现原理**：Redisson 的可重入锁就是基于这个方案实现的。

## 三、缺陷二：不可重试

### 问题描述

获取锁只尝试一次就返回 false，没有重试机制。

### 问题本质

基础 tryLock 是一次性非阻塞实现，高并发场景下抢锁失败就直接终止，无法适配「等待一段时间直到获取锁」的业务场景，资源利用率低。

### 优化方案

**方案 A：自旋重试（基础版）**

设定最大等待超时时间，在超时时间内循环尝试获取锁，每次失败后短暂休眠避免空转消耗 CPU：

```java
public boolean tryLock(long timeoutSec) {
    long deadline = System.currentTimeMillis() + timeoutSec * 1000;
    while (System.currentTimeMillis() < deadline) {
        if (tryLock()) {
            return true;
        }
        Thread.sleep(100);  // 避免空转，可配合指数退避
    }
    return false;
}
```

> **注意**：不是 `while(true)` 死转，要设置超时时间，否则会永远阻塞。

**方案 B：发布订阅（Redisson 方案）**

基于 Redis 发布订阅机制优化：
1. 抢锁失败的线程订阅锁释放频道（`SUBSCRIBE`）
2. 锁被释放时主动发送消息（`PUBLISH`）
3. 等待线程收到消息后尝试加锁

**优势**：减少无效自旋，降低 CPU 与 Redis 压力，同时提升响应速度。

## 四、缺陷三：超时释放

### 问题描述

锁超时释放虽然可以避免死锁，但业务执行耗时较长时会导致锁提前释放，存在安全隐患。

### 问题本质

锁的过期时间是固定预设值，若业务因网络波动、GC、数据库慢查询等原因执行超时：
1. 锁自动过期失效
2. 其他线程趁机抢到锁
3. 两个线程同时执行，引发数据错乱
4. 原线程执行完毕后还可能误删其他线程的锁

### 优化方案：看门狗（Watch Dog）自动续期

**核心机制：**
1. 加锁成功后启动后台守护线程
2. 每隔「锁过期时间的 1/3」周期检查一次
3. 若当前线程仍持有锁，则刷新锁的过期时间
4. 直到业务执行完成主动释放锁
5. 同时设置最大续期时长作为兜底

**Redisson 默认配置：**
- 锁过期时间：30 秒
- 续期间隔：10 秒（30秒的 1/3）
- 最大续期时长：30 分钟（锁过期时间 × 10）

```
时间线：
0s     加锁成功，启动看门狗
10s    第一次续期，重置为30秒
20s    第二次续期，重置为30秒
30s    第三次续期，重置为30秒
...
业务完成  停止续期，释放锁
```

> **兜底机制**：如果业务线程卡死（如死循环），看门狗最多续期 30 分钟后停止，避免锁永久持有。

## 五、缺陷四：主从一致性

### 问题描述

Redis 主从集群存在同步延迟，主节点宕机时，若从节点还未同步锁数据，会出现锁失效。

### 问题本质

Redis 主从复制是异步执行的：
1. 客户端在主节点加锁成功
2. 锁数据还未同步到从节点就发生宕机
3. 从节点被选举为新主节点
4. 锁数据完全丢失
5. 其他客户端可再次成功加锁
6. 最终导致两个线程同时持有锁，互斥性被破坏

```
客户端A → 主节点加锁成功
              ↓
         主节点宕机（数据未同步）
              ↓
从节点升主 → 锁数据丢失
              ↓
客户端B → 新主节点加锁成功 ← 两个客户端同时持有锁！
```

### 优化方案

**方案 A：红锁（RedLock）算法**

部署多个相互独立的 Redis 主节点（通常为 5 个，无主从关系）：

1. 加锁时向所有节点发起请求
2. 超过半数节点加锁成功才算真正获取锁
3. 总耗时小于锁有效期
4. 释放锁时向所有节点执行释放

```
客户端 → 节点1 ✓
       → 节点2 ✓
       → 节点3 ✓（3/5 > 半数，加锁成功）
       → 节点4 ✗
       → 节点5 ✗
```

**争议点**：Martin Kleppmann 质疑 RedLock 不能解决 GC 停顿、时钟跳跃问题，本质上分布式锁的「安全性」和「活性」不可兼得。面试中提到这个争议会加分。

**方案 B：替代方案**

对于一致性要求极高的业务，可改用 ZooKeeper 等 CP 型组件实现分布式锁，牺牲部分性能换取强一致性。

## 六、方案对比总览

| 缺陷 | 问题 | Redisson 解决方案 |
|---|---|---|
| 不可重入 | 同一线程二次加锁失败 | Redis Hash + 重入计数 |
| 不可重试 | 拿不到直接返回 false | 自旋重试 + 发布订阅 |
| 超时释放 | 业务超时锁提前过期 | 看门狗自动续期 |
| 主从一致性 | 主从同步延迟导致锁失效 | 红锁（RedLock） |

## 七、全文总结

- 基础 SETNX 锁只能解决单机互斥，生产环境四大缺陷必须解决
- 可重入用 Hash 结构，可重试用自旋或发布订阅，续期用看门狗，主从用红锁
- Redisson 是 Java 生态最成熟的分布式锁框架，四大缺陷全部有成熟实现
- 生产环境推荐直接使用 Redisson，避免自研实现的逻辑漏洞
