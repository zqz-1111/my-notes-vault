## 二、好友关注功能

### 2.1 关注/取关接口

```java
@PutMapping("/{id}/{isFollow}")
public Result follow(@PathVariable("id") Long followUserId, @PathVariable Boolean isFollow){
    return followService.follow(followUserId, isFollow);
}
```

**Service 实现要点：**
- 防关注自己：`userId.equals(followUserId)` 直接返回失败
- 防重复关注：先 `count()` 判断是否已存在
- MySQL + Redis 双写：关注时 `SADD follow:{userId} followUserId`

```java
// 关注：写入 MySQL + Redis Set
boolean isSuccess = this.save(follow);
if (isSuccess) {
    stringRedisTemplate.opsForSet().add(key, followUserId.toString());
}

// 取关：删除 MySQL + Redis Set
boolean isSuccess = this.remove(...);
if (isSuccess) {
    stringRedisTemplate.opsForSet().remove(key, followUserId.toString());
}
```

## 三、共同关注功能

### 3.1 核心思路

用 Redis Set 的交集操作 `SINTER`，O(N) 复杂度，比查两次 DB 再 Java 取交集高效。

```java
@GetMapping("/common/{id}")
public Result followCommons(@PathVariable("id") Long id){
    return followService.followCommons(id);
}
```

### 3.2 Service 实现

```java
public Result followCommons(Long id) {
    Long userId = UserHolder.getUser().getId();
    String key1 = FOLLOW_KEY + userId;
    String key2 = FOLLOW_KEY + id;
    
    // SINTER 取交集
    Set<String> intersect = stringRedisTemplate.opsForSet().intersect(key1, key2);
    if (Objects.isNull(intersect) || intersect.isEmpty()) {
        return Result.ok(Collections.emptyList());
    }
    
    // 批量查用户信息
    List<Long> ids = intersect.stream().map(Long::valueOf).collect(Collectors.toList());
    List<UserDTO> userDTOList = userService.listByIds(ids).stream()
            .map(user -> BeanUtil.copyProperties(user, UserDTO.class))
            .collect(Collectors.toList());
    return Result.ok(userDTOList);
}
```

### 3.3 为什么用 Redis Set

| 方案 | 做法 | 缺点 |
|------|------|------|
| 查两次 DB | 分别查两人的关注列表，Java 取交集 | 关注人多时查询慢 |
| Redis Set | 关注时写入 Set，共同关注直接 SINTER | 需要维护缓存一致性 |

## 四、关注推送（Feed 流）

### 4.1 三种方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 推模式（写扩散） | 博主发笔记，主动推送到所有粉丝 Feed | 读取快，直接查自己列表 | 大 V 发一次要推几十万次 |
| 拉模式（读扩散） | 用户打开页面，实时查所有关注人的笔记 | 写入快，只写一份 | 关注人多时查询慢 |
| 推拉结合 | 普通博主推模式，大 V 拉模式 | 平衡读写压力 | 实现复杂 |

**本项目用推模式。**

### 4.2 推模式实现流程

1. 博主发笔记 → 保存到 `tb_blog`
2. 查询博主的所有粉丝
3. 遍历粉丝，把笔记 ID 推送到每个粉丝的 Feed 列表
4. 用户打开"关注"页 → 从 Redis 读 Feed 列表
5. 根据笔记 ID 列表查博客详情返回

### 4.3 Redis 数据结构

- Key：`feed:{userId}`
- 类型：Sorted Set（按时间戳排序，支持分页）
- Value：blog ID
- Score：发布时间戳

### 4.4 保存并推送

```java
public Result saveBlog(Blog blog) {
    Long userId = UserHolder.getUser().getId();
    blog.setUserId(userId);
    
    // 1. 保存笔记
    boolean isSuccess = this.save(blog);
    if (!isSuccess) return Result.fail("笔记保存失败");
    
    // 2. 查询笔记作者的所有粉丝
    List<Follow> follows = followService.list(new LambdaQueryWrapper<Follow>()
            .eq(Follow::getFollowUserId, userId));
    
    // 3. 推送到所有粉丝的 Feed
    for (Follow follow : follows) {
        Long id = follow.getUserId();
        String key = FEED_KEY + id;
        stringRedisTemplate.opsForZSet().add(key, blog.getId().toString(), System.currentTimeMillis());
    }
    return Result.ok(blog.getId());
}
```

### 4.5 滚动查询 Feed

**为什么用滚动查询不用传统分页？**

传统分页 `page=2&size=10` 用 OFFSET，新数据插入会导致重复或遗漏。

滚动查询用游标 `lastId=100&size=10`，永远从某个位置往前取，不重复。

```java
public Result queryBlogOfFollow(Long max, Integer offset) {
    Long userId = UserHolder.getUser().getId();
    String key = FEED_KEY + userId;
    
    // ZREVRANGEBYSCORE key Max Min LIMIT offset count
    Set<ZSetOperations.TypedTuple<String>> typedTuples = stringRedisTemplate.opsForZSet()
            .reverseRangeByScoreWithScores(key, 0, max, offset, 5);
    
    if (typedTuples == null || typedTuples.isEmpty()) {
        return Result.ok();
    }
    
    // 解析数据：blogId、minTime、offset
    List<Long> ids = new ArrayList<>(typedTuples.size());
    long minTime = 0;
    int os = 0;
    for (ZSetOperations.TypedTuple<String> tuple : typedTuples) {
        ids.add(Long.valueOf(tuple.getValue()));
        long time = tuple.getScore().longValue();
        if (time == minTime) {
            os++;
        } else {
            minTime = time;
            os = 1;
        }
    }
    
    // 查博客详情，保持 Redis 返回的顺序
    String idStr = StrUtil.join(",", ids);
    List<Blog> blogs = this.list(new LambdaQueryWrapper<Blog>().in(Blog::getId, ids)
            .last("ORDER BY FIELD(id," + idStr + ")"));
    
    // 设置用户信息、是否点赞
    for (Blog blog : blogs) {
        queryUserByBlog(blog);
        isBlogLiked(blog);
    }
    
    // 返回 ScrollResult
    ScrollResult scrollResult = new ScrollResult();
    scrollResult.setList(blogs);
    scrollResult.setOffset(os);
    scrollResult.setMinTime(minTime);
    return Result.ok(scrollResult);
}
```

### 4.6 同分值处理

Sorted Set 按 score（时间戳）排序，同一毫秒可能有多条数据，用 offset 处理：

```
数据：5, 4, 4, 2, 2
最终：minTime=2, os=2（最后一批有2条相同时间戳）
下次查询：从 minTime=2 开始，跳过 os=2 条
```

## 五、踩坑避坑

### 逻辑坑

**坑 1：关注数据不同步**
- 问题：改代码前关注的数据没有写入 Redis
- 后果：共同关注查不到旧数据
- 正确做法：重新关注一次，或者手动补数据

**坑 2：Feed 不包含自己的笔记**
- 问题：saveBlog() 只推给粉丝，没推给自己
- 后果：自己发的笔记在"关注"页看不到
- 解决：`stringRedisTemplate.opsForZSet().add(FEED_KEY + userId, ...)`

**坑 3：Feed 无容量限制**
- 问题：关注人多 + 发帖频繁，Feed 无限增长
- 解决：推送后裁剪 `removeRange(key, 0, -1001)`，只保留最近 1000 条

## 六、全文总结

1. **好友关注**：MySQL 存关注关系 + Redis Set 存一份，方便后续 SINTER
2. **共同关注**：SINTER 取交集，O(N) 复杂度，比查两次 DB 高效
3. **Feed 流推模式**：发笔记时推送到所有粉丝的 Sorted Set，读取快但写入压力大
4. **滚动查询**：用游标不用页码，避免新数据导致重复/遗漏
5. **同分值处理**：Sorted Set 的 offset 机制处理相同时间戳的数据
