# 【秒杀】一人一单与分布式锁

## 一、写在前面

1. 为什么学这个 — 上一篇实现了秒杀下单 + CAS 防超卖，但没防重复下单。用户可以反复点抢购，一个人下 100 单。"一人一单"就是解决这个问题的。
2. 学完你能掌握什么 — 单机用 `synchronized` 实现一人一单，理解为什么集群下要换 Redis 分布式锁，以及分布式锁的核心原理。

## 二、一人一单的需求

同一个用户抢同一张秒杀券，只能下一次单。逻辑上很简单——下单前查一下该用户是否已经买过这张券。

但问题出在**并发**上。

## 三、单机方案：synchronized + intern()

### 3.1 为什么普通写法不防并发？

```java
// 两个请求同时进来，都查到 count=0，都通过校验，重复下单
int count = query().eq("user_id", userId).eq("voucher_id", voucherId).count();
if (count > 0) {
    return Result.fail("已购买过");
}
// ... 创建订单
```

`@Transactional` 只保证数据库事务原子性，不防并发读。两个事务可以同时读到 count=0。

### 3.2 用 synchronized 串行化

把同一用户的请求锁住，让它们排队执行：

```java
Long userId = UserHolder.getUser().getId();
synchronized (userId.toString().intern()) {
    // 查重 + 扣库存 + 创建订单，串行执行
}
```

### 3.3 为什么要 intern()？

`synchronized` 锁的是**对象引用**，不是值。

| 写法 | 问题 |
|---|---|
| `synchronized(userId)` | Long 对象，`==` 比较引用，值相同但对象不同，锁不住 |
| `synchronized(userId.toString())` | 每次 `new String()`，还是不同的锁 |
| `synchronized(userId.toString().intern())` ✅ | `intern()` 返回常量池里的同一个对象，同一 userId 锁同一把 |

### 3.4 为什么要拆成两个方法？

```java
// 外层：加锁
@Override
public Result seckillVoucher(Long voucherId) {
    ...
    synchronized (userId.toString().intern()) {
        IVoucherOrderService proxy = (IVoucherOrderService) AopContext.currentProxy();
        return proxy.createVoucherOrder(voucherId);
    }
}

// 内层：事务
@Transactional
public Result createVoucherOrder(Long voucherId) {
    // 查重 + 扣库存 + 创建订单
}
```

**原因：`@Transactional` 靠 AOP 代理生效。**

同一个类里 `this.method()` 调用不走代理，事务白写。必须通过 `AopContext.currentProxy()` 拿到代理对象调用。

```java
// ❌ 不走代理，事务不生效
this.createVoucherOrder(voucherId);

// ✅ 走代理，事务生效
IVoucherOrderService proxy = (IVoucherOrderService) AopContext.currentProxy();
proxy.createVoucherOrder(voucherId);
```

> 使用 `AopContext.currentProxy()` 需要两个前提：
> 1. 启动类加 `@EnableAspectJAutoProxy(exposeProxy = true)`
> 2. pom.xml 加 `aspectjweaver` 依赖

### 3.5 整体执行流程

```
线程A（用户1001的第1个请求）:
  获取锁(userId=1001) → 查count=0 → 扣库存 → 创建订单 → 提交事务 → 释放锁 ✅

线程B（用户1001的第2个请求，同时到达）:
  获取锁(userId=1001) → 阻塞等待...
  → 拿到锁 → 查count=1 → "已购买过" → 释放锁 ❌

线程C（用户2002的请求，同时到达）:
  获取锁(userId=2002) → 查count=0 → 扣库存 → 创建订单 → 提交事务 → 释放锁 ✅
  （和线程A并行，因为是不同的锁）
```

**效果：同一用户串行，不同用户并行。**

### 3.6 局限性

`synchronized` 是 **JVM 级别**的锁——每个 JVM 各有一把，互不知道对方的存在。

```
单机部署（1个JVM）✅
┌──────────────┐
│    JVM-1     │
│ 线程A  线程B │ ← 同一把锁，串行
└──────────────┘

集群部署（多个JVM）❌
┌──────────────┐  ┌──────────────┐
│    JVM-1     │  │    JVM-2     │
│    线程A     │  │    线程C     │ ← 各锁各的，拦不住
└──────────────┘  └──────────────┘
```

## 四、集群方案：Redis 分布式锁

### 4.1 核心思路

找一个**所有 JVM 共享的东西**当锁。Redis 就是最合适的选择——所有应用服务器都连同一个 Redis。

```
JVM-1: SET lock:order:1001:5 NX EX 10  → 成功，拿到锁
JVM-2: SET lock:order:1001:5 NX EX 10  → 失败（key已存在），等着
```

### 4.2 加锁：SET key NX EX

```bash
SET lock:order:用户id:优惠券id {UUID} NX EX 10
```

| 参数 | 作用 |
|---|---|
| `NX` | 只有 key 不存在时才能设置 → 互斥 |
| `EX 10` | 10秒后自动过期 → 防死锁 |
| `{UUID}` | 当前线程的唯一标识 → 防误删 |

### 4.3 为什么要 UUID？防锁误删

```
线程A: 拿到锁 → 业务执行太慢 → 锁过期自动释放 → 删锁
线程B:          A的锁没了 → 拿到锁 → 执行业务
线程C:                                    A把B的锁删了！❌
```

加了 UUID，删锁前先比对：

```
线程A: 想删锁 → GET value → 不是自己的UUID → 不删 ✅
线程B: 继续正常执行
```

### 4.4 为什么要 Lua 脚本？原子删锁

比对 UUID 和删 key 是两步操作，中间可能被打断：

```lua
-- Lua 脚本：判断 + 删除一步完成，不会被打断
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
end
return 0
```

Redis 执行 Lua 脚本是单线程的，保证原子性。

### 4.5 锁续期：看门狗机制

业务还没执行完，锁就过期了怎么办？

- **手写方案：** 起后台线程定时续期，实现复杂
- **Redisson 方案：** 内置看门狗，每隔 1/3 过期时间自动检查续期，一行代码搞定

```java
RLock lock = redissonClient.getLock("lock:order:1001:5");
lock.lock();   // 自动续期，业务完了自动释放
```

### 4.6 完整流程

```
┌─────────────────────────────────────────────────┐
│  线程A 请求下单                                  │
├─────────────────────────────────────────────────┤
│  1. SET lock:order:1001:5 {uuid} NX EX 10       │
│     → 成功，拿到锁                                │
│                                                  │
│  2. 执行业务（查重 + 扣库存 + 下单）               │
│                                                  │
│  3. Lua脚本：比对uuid → 是自己的 → DEL            │
│     → 释放锁                                      │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│  线程B 同时请求（同一用户同一券）                   │
├─────────────────────────────────────────────────┤
│  1. SET lock:order:1001:5 {uuid} NX EX 10       │
│     → 失败（key已存在）                           │
│                                                  │
│  2. 等待 / 重试 / 返回"稍后再试"                   │
└─────────────────────────────────────────────────┘
```

## 五、分布式锁方案对比

| 维度 | MySQL | Redis | Zookeeper |
|---|---|---|---|
| **互斥** | 唯一索引/行锁，阻塞式 | SETNX，非阻塞 | 临时有序节点，天然强互斥 |
| **高可用** | 依赖主从架构，单点风险高 | 主从/哨兵/集群，较好 | ZAB 协议，过半存活即可 |
| **高性能** | 最差，磁盘 IO | 最优，纯内存操作 | 中等，节点创建删除有开销 |
| **安全性** | 事务回滚自动释放 | 需 Lua+UUID+看门狗补强 | 临时节点断开自动删除，最安全 |
| **适用场景** | 低并发、已有数据库 | 高并发短任务（秒杀） | 强一致性、分布式协调 |

**结论：互联网项目主流选 Redis + Redisson，性能够高、接入够简单。**

## 六、踩坑避坑指南

### 踩坑 1：createVoucherOrder 报红"无法解析方法"

- 问题：`AopContext.currentProxy()` 返回的是接口类型，但 `createVoucherOrder` 只在实现类里定义了
- 后果：编译报红
- 解决：在 `IVoucherOrderService` 接口里补上方法声明

### 踩坑 2：aspectjweaver 不指定版本导致 JVM Target 17 报错

- 问题：Spring Boot 2.3.12 的依赖管理拉到了需要 JDK 17 的新版 aspectjweaver
- 后果：`java: 无法编译为 JVM 目标 17 配置的模块`
- 解决：手动指定版本 `1.9.7`，兼容 JDK 8

### 踩坑 3：intern() 的作用不理解

- 问题：`synchronized(userId.toString())` 以为能锁住同一用户
- 后果：每次 `toString()` 都是新对象，锁不住
- 正确理解：`intern()` 把字符串放入常量池，保证同一内容返回同一引用

## 七、全文总结

- 一人一单的核心：下单前查重（userId + voucherId），保证同一用户同一券只下一单
- 单机用 `synchronized + intern()` 按 userId 加锁，同一用户串行执行
- `@Transactional` 通过 AOP 代理生效，同类内部调用需要用 `AopContext.currentProxy()` 拿代理
- 集群下 synchronized 不够，需要 Redis 分布式锁（SETNX + UUID + Lua 脚本 + 看门狗）
- 生产环境直接用 Redisson 框架，不用手写
