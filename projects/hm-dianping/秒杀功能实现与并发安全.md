# 【秒杀】秒杀功能实现与并发安全

## 一、写在前面

1. 为什么学这个 — 秒杀是电商的核心场景，高并发下如何安全地扣减库存、防止超卖，是后端开发必须掌握的技能
2. 学完这篇你能掌握什么 — 秒杀优惠券的完整下单链路，以及用乐观锁 CAS 方案解决超卖问题

## 二、前置准备

- 数据库三张表：`tb_voucher`（优惠券基本信息）、`tb_seckill_voucher`（秒杀专属信息：库存、时间）、`tb_voucher_order`（订单表）
- Redis 已预热秒杀库存到 `seckill:stock:{voucherId}`
- 前端页面 `shop-detail.html` 已有秒杀券 UI（通过 `v.type` 区分普通券和秒杀券）

## 三、秒杀下单完整链路

### 3.1 前端触发（shop-detail.html）

用户点击"限时抢购"按钮，前端先做一层校验（体验优化，不是安全防线）：

```javascript
seckill(v){
    // 1. 登录检查
    if(!token){ this.$message.error("请先登录"); return; }
    // 2. 时间检查
    if(this.isNotBegin(v)){ this.$message.error("秒杀尚未开始！"); return; }
    if(this.isEnd(v)){ this.$message.error("秒杀已经结束！"); return; }
    // 3. 库存检查
    if(v.stock < 1){ this.$message.error("库存不足！"); return; }
    // 4. 发请求
    axios.post("/voucher-order/seckill/" + id)
    .then(({data}) => {
        this.$message.success("抢购成功，订单id：" + data)
    })
}
```

> ⚠️ 前端校验只是体验优化，真正防并发靠后端。用户可以用 Postman 绕过前端直接调接口。

### 3.2 Controller 转发

```java
@PostMapping("seckill/{id}")
public Result seckillVoucher(@PathVariable("id") Long voucherId) {
    return voucherOrderService.seckillVoucher(voucherId);
}
```

### 3.3 Service 核心逻辑（VoucherOrderServiceImpl）

```java
@Override
@Transactional  // 保证扣库存+创建订单的原子性
public Result seckillVoucher(Long voucherId) {
    // 1. 查询秒杀券信息
    SeckillVoucher voucher = seckillVoucherService.getById(voucherId);

    // 2. 判断秒杀是否开始
    if (voucher.getBeginTime().isAfter(LocalDateTime.now())) {
        return Result.fail("秒杀尚未开始！");
    }

    // 3. 判断秒杀是否结束
    if (voucher.getEndTime().isBefore(LocalDateTime.now())) {
        return Result.fail("秒杀已经结束！");
    }

    // 4. 判断库存是否充足
    if (voucher.getStock() < 1) {
        return Result.fail("库存不足！");
    }

    // 5. CAS 扣减库存（乐观锁）
    boolean success = seckillVoucherService.update(new LambdaUpdateWrapper<SeckillVoucher>()
            .eq(SeckillVoucher::getVoucherId, voucherId)
            .gt(SeckillVoucher::getStock, 0)   // stock > 0 才扣
            .setSql("stock = stock - 1"));
    if (!success) {
        return Result.fail("库存不足！");
    }

    // 6. 创建订单
    VoucherOrder voucherOrder = new VoucherOrder();
    long orderId = redisIdWorker.nextId("order");
    voucherOrder.setId(orderId);
    Long userId = UserHolder.getUser().getId();
    voucherOrder.setUserId(userId);
    voucherOrder.setVoucherId(voucherId);
    save(voucherOrder);

    return Result.ok(orderId);
}
```

### 3.4 为什么要加 @Transactional？

扣库存和创建订单是两个数据库操作，必须保证原子性：

| 场景 | 不加事务 | 加事务 |
|---|---|---|
| 扣库存成功，创建订单失败 | 库存凭空消失 ❌ | 回滚，库存不变 ✅ |
| 两个都成功 | 正常 | 正常 |

## 四、超卖问题详解

### 4.1 什么是超卖？

库存只剩 1 张券，两个用户同时点抢购：

```
用户A请求 → 读库存(stock=1) → 判断"充足" → 扣库存(stock=0) → 创建订单 ✅
                                      ↑
用户B请求 → 读库存(stock=1) → 判断"充足" → 扣库存(stock=-1) → 创建订单 ✅
              ↑
          这里读到的还是1！A还没扣完
```

结果：库存只有 1 张，卖出去了 2 张。这就叫**超卖**。

### 4.2 为什么先查后改会超卖？

```java
// 第1步：读
SeckillVoucher voucher = seckillVoucherService.getById(voucherId);
if (voucher.getStock() < 1) { return fail; }

// ← 这个间隙里，另一个请求也读到了 stock=1

// 第2步：写
seckillVoucherService.update().setSql("stock = stock - 1")...
```

读和写不是原子操作，中间有时间差，别的请求可能在这个间隙读到同样的值。

## 五、解决方案对比

### 5.1 悲观锁

- 核心思想：**默认冲突一定会发生**，操作前先加锁，强制串行执行
- 实现：`SELECT ... FOR UPDATE`、`synchronized`、`Lock`
- 优点：强一致性，不会超卖
- 缺点：串行执行，性能差，高并发下大量请求阻塞
- 适用：写多读少、冲突概率高（金融转账、票务扣减）

### 5.2 乐观锁

- 核心思想：**默认冲突很少发生**，全程不加锁，提交时校验数据是否被改过
- 实现：版本号机制、CAS（Compare-And-Swap）
- 优点：无锁阻塞，并发性能好
- 缺点：冲突频繁时大量重试，反而降低性能
- 适用：读多写少、冲突概率低

### 5.3 我们用的 CAS 方案

```java
boolean success = seckillVoucherService.update(new LambdaUpdateWrapper<SeckillVoucher>()
        .eq(SeckillVoucher::getVoucherId, voucherId)
        .gt(SeckillVoucher::getStock, 0)   // WHERE stock > 0
        .setSql("stock = stock - 1"));
```

生成的 SQL：
```sql
UPDATE tb_seckill_voucher SET stock = stock - 1 WHERE voucher_id = ? AND stock > 0
```

- `stock > 0` 保证了库存不会变成负数
- 如果扣减失败（返回 false），说明库存已经没了
- 本质是"条件扣减"，是 CAS 的变体

### 5.4 方案选型

| 方案 | 思路 | 适合场景 |
|---|---|---|
| 悲观锁 | 先加锁再操作 | 写多读少、冲突激烈 |
| 乐观锁 CAS | 提交时校验 | 读多写少、冲突较少 |
| Redis + Lua | Redis 层面一步完成 | 高并发秒杀（后面学） |

> 秒杀场景冲突率很高，乐观锁会有大量请求重试失败。更优方案是 Redis + Lua 脚本，在 Redis 层面原子性地完成判断 + 扣减。

## 六、踩坑避坑指南

### 踩坑 1：接口没接通

- 问题：Controller 写死返回 `Result.fail("功能未完成")`，没有调用 Service
- 后果：前端怎么点都是"功能未完成"
- 解决：注入 Service，调用实际方法

### 踩坑 2：接口方法没声明

- 问题：`VoucherOrderServiceImpl` 写了 `@Override` 的方法，但 `IVoucherOrderService` 接口里没声明
- 后果：编译报错
- 解决：在接口里补上 `Result seckillVoucher(Long voucherId);`

### 踩坑 3：前端看不到秒杀券

- 问题：数据库里 `tb_seckill_voucher` 表没有数据，或者 `tb_voucher.type` 不是 1
- 后果：前端只显示普通券
- 解决：插入 type=1 的券 + 对应的秒杀信息 + Redis 库存

### 踩坑 4：@Transactional 不能防超卖

- 问题：以为加了 `@Transactional` 就安全了
- 后果：事务保证的是原子性，不保证并发安全。两个事务可以同时读到 stock=1
- 正确理解：`@Transactional` 防的是"扣了库存但订单没创建"，不防超卖

## 八、全文总结

- 秒杀链路：前端校验 → 查券 → 判时间 → CAS 扣库存 → 创建订单
- `@Transactional` 保证扣库存+创建订单的原子性，但不能防超卖
- 超卖根因：先查后改不是原子操作，读写之间有时间差
- 乐观锁 CAS：通过 `WHERE stock > 0` 条件扣减，防止库存变负数
- 高并发秒杀更优方案：Redis + Lua（后面学）
