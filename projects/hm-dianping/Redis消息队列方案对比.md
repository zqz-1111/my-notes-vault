# 【Redis】消息队列方案对比（Pub/Sub vs List vs Stream）

## 一、写在前面

1. 为什么学这个 — 秒杀异步下单需要消息队列，Redis 提供了三种方案，各有适用场景，选错了要么丢消息要么重复消费
2. 学完这篇你能掌握 — Pub/Sub、List、Stream 三种方案的原理、区别、适用场景，以及为什么秒杀场景选 Stream

## 二、先搞清楚：什么是消息队列

**消息队列 = 生产者发消息、消费者收消息的中间存储。**

生活类比：
- **Pub/Sub**：电台广播，播完就没了，听众不在就错过，所有听众听到同一内容
- **List**：快递柜，放进去什么时候取都行，取走就没了，只能一个人取
- **Stream**：带签收单的快递柜，取了要签字确认，没签字的可以找回，还能多人分工取

## 三、方案一：Pub/Sub（发布订阅）

### 3.1 核心命令

```bash
SUBSCRIBE channel1        # 订阅频道
PUBLISH channel1 "hello"  # 往频道发消息
```

### 3.2 工作原理

```
发布者 ──PUBLISH──→ [channel: order] ──→ 订阅者A（收到）
                                      ──→ 订阅者B（收到）
                                      ──→ 订阅者C（收到）
```

广播模式——一条消息所有订阅者都能收到，类似微信公众号推送。

### 3.3 特性

- ✅ 简单，两个命令搞定
- ✅ 支持多频道
- ❌ **不持久化**——发完就没了，订阅者不在线就错过
- ❌ **没有 ACK**——不知道消费者有没有收到
- ❌ **广播模式**——每个订阅者都收到一份，不能竞争消费

### 3.4 适用场景

- 实时通知（"有新订单了"广播给所有后台）
- 配置变更通知
- 简单的事件总线

### 3.5 为什么不适合秒杀

秒杀需要**竞争消费**——100 个订单，谁抢到谁处理，处理完就没了。Pub/Sub 是广播，100 个订单每个消费者都收到一份，那不就重复下单了？

## 四、方案二：List（列表）

### 4.1 核心命令

```bash
LPUSH order "msg1"    # 左边推入
LPUSH order "msg2"
BRPOP order 30        # 右边阻塞弹出，最多等30秒
```

### 4.2 工作原理

```
生产者 ──LPUSH──→ [List: order] ──BRPOP──→ 消费者A
                     msg3                  消费者B（等A取完再取）
                     msg2
                     msg1
```

FIFO（先进先出），消息被取出就从 List 中删除。

### 4.3 特性

- ✅ 持久化——消息存在 Redis 里，重启不丢
- ✅ 阻塞读取——`BRPOP` 没消息时阻塞等待，不浪费 CPU
- ✅ 简单——两个命令搞定
- ❌ **没有 ACK**——取走就没了，消费者崩了消息就丢了
- ❌ **没有消费者组**——不能多个消费者分摊压力
- ❌ **不能回溯**——取走就从 List 删除了

### 4.4 适用场景

- 简单的异步任务队列
- 不要求高可靠性的场景
- 单消费者模式

### 4.5 为什么不是最优解

List 的问题：`BRPOP` 取走就没了。如果消费者取到消息后崩了，这个订单就丢了。没有确认机制，也没有消费者组。

## 五、方案三：Stream（流）

### 5.1 核心命令

```bash
# 生产者：往队列里塞消息
XADD order * userId 1001 voucherId 1 orderId 123456
#         ↑ 消息ID（*自动生成）    ↑ 消息内容（key-value键值对）

# 消费者组模式（竞争消费）
XGROUP CREATE order g1 0        # 创建消费者组 g1
XREADGROUP GROUP g1 c1 COUNT 1 BLOCK 2000 STREAMS order >
#          组名  消费者名           > 表示只取新消息

# 确认消息
XACK order g1 消息ID
```

### 5.2 工作原理

```
                    ┌→ 消费者A（处理消息1,3）
消息 → [Stream] → 组g1 → 消费者B（处理消息2,4）
                    └→ 消费者C（处理消息5）
```

每个消息只被组内一个消费者处理，不会重复消费。

### 5.3 核心特性

**① 消费者组（Consumer Group）**

多个消费者分摊压力——这就是集群场景下需要的。

**② ACK 确认机制**

```
XREADGROUP → 拿到消息 → 处理成功 → XACK → Redis 才删掉
                                      ↓
                                没ACK？下次还能拿到（防丢）
```

List 的 `BRPOP` 取走就没了，Stream 有 `XPENDING` 可以查未确认的消息，重新分配给其他消费者。

**③ 消息 ID 天然有序**

```
1718956800000-0
1718956800000-1
1718956800001-0
```

时间戳 + 序号，可以按 ID 范围查询历史消息，方便排查问题。

**④ Pending 列表兜底**

消费者拿到消息但没 ACK（比如崩了），消息会进入 pending 列表。下次可以从 pending 重新读取，不会丢。

### 5.4 适用场景

- 正式的生产环境消息队列
- 需要高可靠性的异步任务
- 多消费者集群分摊压力
- 需要消息回溯和排查

## 六、三方案完整对比

| 特性 | List | Pub/Sub | Stream |
|---|---|---|---|
| 持久化 | ✅ | ❌ | ✅ |
| 消息确认（ACK） | ❌ 取走就没了 | ❌ | ✅ `XACK` |
| 消费者组 | ❌ | ❌ | ✅ |
| 阻塞读取 | ✅ `BRPOP` | ✅ 自动 | ✅ `BLOCK` |
| 历史消息回溯 | ❌ | ❌ | ✅ 按 ID 范围查 |
| 消息堆积 | ✅ | ❌ 不在线就丢 | ✅ |
| 复杂度 | 简单 | 简单 | 稍复杂 |
| 适用场景 | 简单队列 | 广播通知 | 正式消息队列 |

## 七、选型决策树

```
需要消息队列？
├─ 不需要持久化、广播模式 → Pub/Sub
└─ 需要持久化
   ├─ 单消费者、不关心丢消息 → List
   └─ 需要消费者组/ACK/高可靠 → Stream
```

**秒杀场景选 Stream**：需要消费者组（集群竞争消费）+ ACK 确认（防丢）+ 持久化（重启不丢）。

## 八、一句话总结

> Pub/Sub 是"电台广播"，播完就没了，听众不在就错过；List 是"快递柜"，放进去取走就没了；Stream 是"带签收单的快递柜"，取了要确认，没确认还能找回，还能多人分工取。
