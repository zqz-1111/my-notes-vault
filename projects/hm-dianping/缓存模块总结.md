## 三、核心知识全景图

### 3.1 Cache Aside 模式（缓存读写基础）

```
读请求：  先查 Redis → 命中返回 → 未命中查 MySQL → 写入 Redis → 返回
写请求：  先更新 MySQL → 再删除 Redis 缓存
```

**关键决策：**
- 为什么是「删缓存」而不是「更新缓存」？→ 避免并发写入导致脏数据
- 为什么是「先改库后删缓存」？→ 先删缓存会导致其他线程读到旧值并写入
- 为什么需要 TTL？→ 兜底防止永久脏数据

### 3.2 缓存三大问题

| 问题 | 核心诱因 | 数据是否存在 | 影响范围 | 解决方案 |
|---|---|---|---|---|
| **穿透** | 查询不存在的数据 | 缓存和数据库都没有 | 持续打库 | 缓存空对象、布隆过滤器 |
| **雪崩** | 大批 key 同时过期 / Redis 宕机 | 数据存在 | 全局大面积 | TTL 加随机值、集群、限流 |
| **击穿** | 热点 key 突然过期 | 数据库存在，缓存临时失效 | 单个热点 | 互斥锁、逻辑过期 |

**记忆口诀：** 穿透 = 东西不存在穿过去了，击穿 = 盾牌被打穿一个点，雪崩 = 整个盾牌塌了

### 3.3 缓存击穿的三种解法

| 解法 | 是否阻塞 | 数据一致性 | 适用场景 |
|---|---|---|---|
| **互斥锁** | 是（等待） | 强一致 | 一致性要求高、并发中等 |
| **逻辑过期** | 否（返回旧数据） | 短暂不一致 | 高并发、允许短暂延迟 |
| **永不过期** | 否 | 取决于更新频率 | 核心热点数据 |

## 四、CacheClient 工具类设计

### 4.1 为什么封装

之前每个 Service 方法里都是一大坨 Redis 操作代码，封装成工具类后一行调用搞定，复用性拉满。

### 4.2 四个核心方法

```
方法1 set()
  作用：存数据 + TTL
  场景：普通缓存写入
  原理：JSONUtil.toJsonStr(value) → Redis SET + TTL

方法2 setWithLogicalExpire()
  作用：存数据 + 逻辑过期时间
  场景：配合方法4防击穿
  原理：包装成 RedisData（data + expireTime），JSON 序列化后存入 Redis
  关键：Redis key 本身永不过期，靠 value 里的 expireTime 判断

方法3 handleCachePenetration()
  作用：查询 + 缓存空值防穿透
  场景：普通数据查询
  流程：查 Redis → 命中返回 → 命中空值返回 null → 未命中查 DB → DB 没有缓存空值 → DB 有则写缓存返回

方法4 handleCacheBreakdown()
  作用：查询 + 逻辑过期防击穿
  场景：热点数据查询
  流程：查 Redis → 未命中返回 null → 命中判断逻辑过期 → 未过期返回 → 过期拿锁 → 拿到锁开子线程重建 + 返回旧数据 → 没拿到锁双检缓存 → 返回旧数据
```

### 4.3 辅助组件

- **RedisData**：逻辑过期包装类，包含 `data`（实际数据）和 `expireTime`（逻辑过期时间）
- **tryLock / unlock**：基于 `SETNX` 的简单互斥锁，TTL 10 秒防死锁
- **CACHE_REBUILD_EXECUTOR**：10 线程的线程池，用于异步重建缓存

## 五、方案选型总结

### 什么时候用哪个方法？

```
普通数据（非热点）→ handleCachePenetration（缓存空值防穿透）
热点数据（高并发）→ handleCacheBreakdown（逻辑过期防击穿）
预热/更新缓存    → setWithLogicalExpire（写入逻辑过期数据）
简单缓存写入     → set（基础 TTL 缓存）
```

### 实际使用流程

```
1. 项目启动时：用 setWithLogicalExpire() 预热热点数据到 Redis
2. 查询热点数据：用 handleCacheBreakdown() 查询
3. 查询普通数据：用 handleCachePenetration() 查询
4. 数据库数据变了：用 setWithLogicalExpire() 更新缓存
```

## 六、踩坑汇总

| 坑 | 问题 | 解决 |
|---|---|---|
| setWithLogicalExpire 存错对象 | 存了原始 value 而不是 RedisData | 改为 `JSONUtil.toJsonStr(redisData)` |
| 方法名大小写写错 | Java 大小写敏感，`trylock` ≠ `tryLock` | 保持一致 |
| 缓存空对象 TTL 设太长 | 数据库新增数据但缓存还是空值 | TTL 设 2~5 秒 |
| 互斥锁忘记释放 | 异常退出导致死锁 | try-finally 确保释放 |
| TTL key 拼写错误 | 刷新的是原始 token 而不是带前缀的 key | 统一用 `tokenKey` |
