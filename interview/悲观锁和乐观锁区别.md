# 悲观锁和乐观锁有什么区别？

## 问题

悲观锁和乐观锁有什么区别？分别适用于什么场景？

## 答案

**悲观锁：** 默认并发冲突一定会发生，操作数据前先加锁，强制同一时间只有一个线程能操作。实现：`synchronized`、`Lock`、`SELECT ... FOR UPDATE`。优点是强一致性，缺点是性能差、高并发下大量请求阻塞。适合写多读少、冲突概率高的场景（金融转账、票务扣减）。

**乐观锁：** 默认并发冲突很少发生，全程不加锁，提交时校验数据是否被改过。实现：版本号机制（`WHERE version = 旧版本`）、CAS 算法。优点是并发性能好，缺点是冲突频繁时大量重试。适合读多写少、冲突概率低的场景。

## 深入追问

- 追问 1：CAS 是什么？ — Compare-And-Swap，比较并交换，无锁算法。比较内存值与预期值是否一致，一致则更新，是 Java 并发包的底层基础
- 追问 2：乐观锁在秒杀场景有什么问题？ — 秒杀冲突率很高，大量请求会重试失败，性能反而更差。更优方案是 Redis + Lua 脚本
- 追问 3：@Transactional 能防超卖吗？ — 不能。事务保证的是原子性（要么都成功要么都回滚），不保证并发安全。两个事务可以同时读到相同的库存值
